## Phase 2b — Typed plain-CRUD household records + cascade graph

**Review:** approved=True · confidence=high

### Blueprint
**goal:** Land the typed CKRecord encode/decode layer + the CASCADE/SET-NULL reference graph for every household-scoped plain-CRUD record onto the existing Phase 2a HouseholdSyncEngine + HouseholdLocalStore, as INERT LWW pass-through. 2b establishes the typed-record codec pattern (recordName policy + field-type mapping + ref kind) that Phase 3 (CKAsset) and Phase 4 (grocery/event field-merge) extend; it must NOT add any merge logic. Deliver headless-unit-testable codec/cascade logic plus a DEBUG sim panel check proving one typed record (Recipe + a CASCADE child + a SET-NULL self-ref) round-trips and orphan-sweeps through the real CKSyncEngine.

**approach:** Mirror the existing module split. GroceryMerge already proves the idiom: pure Swift value structs (Mergeable) in a CloudKit-free target, unit-tested headlessly, with the thin CKRecord adapter deferred to the wiring phase. RecordNames already centralizes the irreversible DET key builders. HouseholdSyncEngine/HouseholdLocalStore already own the stack with named override seams.

Add a NEW target `HouseholdRecords` (CloudKit-free, like GroceryMerge) holding: (1) a per-type field/recordName/reference manifest as pure Swift, and (2) a CKRecord codec behind `#if canImport(CloudKit)` (like HouseholdZoneProvisioner/HouseholdLocalStore guard CloudKit). Reasons to keep the manifest CloudKit-free: it is the part with real branching logic (Bool→INT64, date→TIMESTAMP, dual-typed catalog refs, cascade-vs-set-null classification, recordName derivation) and it must unit-test headlessly the way RecordNamesTests and ResolverTests do. The codec layer is mechanical glue tested on-sim.

Three things to build:

1. RECORDNAME builders — extend the existing `RecordNames` enum in CloudKitProvisioning (do NOT fork it). It already has householdSetting/termAlias/eventAttendee/eventPantrySupplement/recipeImage. 2b adds nothing new structurally for the PK types (recordName == legacy id, passed through verbatim) but should add a `passthroughPK(_ id:)` doc marker and unit-pin that DET junction keys are stable. The hazard to encode in a test: EventAttendee DET key order is FIXED (`<eventID>_<guestID>`) — swapping halves forks the junction.

2. FIELD CODEC + REFERENCE GRAPH — a `HouseholdRecordType` enum (one case per 2b record type: householdProfile, householdSetting, householdTermAlias, recipe, recipeIngredient, recipeStep, guest, event, eventAttendee, eventMeal, eventMealIngredient, weekChangeBatch, weekChangeEvent, feedbackEntry, baseIngredient, ingredientVariation) carrying: its recordName policy (PK|DET), its parent CKReference field name (the ONE .deleteSelf edge, or none), and its SET-NULL ref/String-key fields. The encode/decode functions map the value struct ↔ CKRecord field bag applying the type rules. Mirror the production SQLAlchemy models (app/models/recipe.py, event.py, catalog.py, week.py, aliases.py, household_setting.py) field-for-field — read them; do not transcribe from the research JSON without re-checking, since CloudKit-schema field names are irreversible.

Type-mapping rules (encode side): Bool→INT64 0/1 (mirror the deployed `active INT64` convention in phase0-schema.ckdb PreferenceSignal/IngredientPreference); Python date / DateTime→CKRecord Date (TIMESTAMP); Float→Double; Int→Int64; Text/String→String; JSON-blob columns (override_payload_json, constraint_coverage, reason_codes)→String pass-through. DROP every household_id and user_id column (zone identity replaces them — there is NO household_id field on any record). Optional Python columns stay optional (absent key, not empty value).

Reference-kind classification (the load-bearing, irreversible part):
- CASCADE parent → encode as a CKReference with action `.deleteSelf`. The engine must STILL enqueue child deletes locally (CloudKit's .deleteSelf only fires on the issuing device; orphan-sweeping a fetched parent-delete is the client's job). This needs a new HouseholdSyncEngine surface (see #3).
- SET-NULL → plain CKReference (action `.none`) when the target lives in the SAME household zone, OR a plain String key when the target is cross-DB PUBLIC (catalog base_ingredient_id / ingredient_variation_id / recipe_template_id / merged_into_id chains). The codec picks per-target at encode time; a dangling ref nulls locally on fetch — never crash.
- Self-ref split is a NAMED hazard to lock with tests: Recipe.base_recipe_id = SET-NULL (plain ref); RecipeStep.parent_step_id = CASCADE (.deleteSelf); BaseIngredient/IngredientVariation.merged_into_id = SET-NULL WALK (follow chain to nil, cap depth, guard cycles). Swapping the first two deletes recipe variants when a base is removed.
- EventAttendee→Guest MISMATCH must be resolved before encoding: Postgres has guest_id ondelete=CASCADE, but only ONE parent CKReference per record can be .deleteSelf and eventRef is that parent. So guestRef must be plain (client-swept on Guest delete). Flag this as an OpenQuestion for the engineer to confirm intended semantics, defaulting to eventRef=.deleteSelf + guestRef=plain.

3. ENGINE CASCADE SWEEP — add an orphan-sweep API to HouseholdSyncEngine.delete (or a sibling `deleteCascading`). Today `delete(_ recordID:)` just removes one record + enqueues one delete. For a CASCADE parent, the client must enqueue deletes for every locally-known child (records whose parent-ref points at the deleted id). Mirror the existing `delete` idiom but iterate the local store. This is the ONE engine change 2b needs; keep applyRemoteModification and handleFailedSave UNTOUCHED (they stay blanket LWW — 2b records are inert pass-through; Phase 4 swaps the merge in at those exact seams). Add a child-lookup helper to HouseholdLocalStore (scan records for a parent-ref field value) mirroring its existing lock+scan accessors.

The CKDSL: append the 2b record types to .docs/ai/phases/phase0-schema.ckdb following the EXACT DSL idiom already there (RECORD TYPE Name ( field TYPE [QUERYABLE|SORTABLE], GRANT WRITE TO _creator, GRANT READ,CREATE TO _icloud )). Parent CKReference fields are implicitly queryable — do not double-mark. This file is re-imported cumulatively; the user runs validate-schema + the deploy (needs the CloudKit management token) — do NOT attempt to deploy from the agent thread.

## Phase 2c — CKShare invite/accept + owner/participant + single-household

### Blueprint
