# SP-C — AI track, slice AI-4: AI recipe images

> 2026-06-22. Fourth AI slice. Image generation via BYO key → CKAsset on the recipe. The save path
> (RecipeRepository.setImage / RecipeImageCodec) is DONE — this slice adds the image-gen + the rewire.

## 0. Goal + scope
Bring the recipe image-gen methods off Fly: `regenerateRecipeImage(recipeID:)` (one recipe) and
`backfillRecipeImages()` (all recipes missing an image). Generate via the user's BYO key (OpenAI
`gpt-image-1` or Gemini `gemini-2.5-flash-image-preview`), per the `image_provider` setting, → image
bytes → `RecipeRepository.setImage` (CKAsset). No Fly.

## 1. Design decisions (Lead, from the map)
- **Image-gen via BYO key, image-provider-selected.** `image_provider` (private-plane profile, openai|gemini)
  picks the API. The KEY: for OpenAI images reuse the OpenAI Keychain key (same key as text); for Gemini a
  GEMINI Keychain key (the user adds one — Anthropic has no image-gen, so Anthropic-text users need an
  OpenAI or Gemini key for images). Reuse KeychainKeyStore (CloudModel.openAI/.gemini).
- **Failover OpenAI→Gemini** (port `recipe_image_ai.generate_recipe_image`): if image_provider==openai and
  the call is a TRANSIENT error (5xx/429/network) AND a Gemini key exists → retry once via Gemini. Gemini
  primary → no failover. Permanent errors (4xx/auth) → surface.
- **Prompt = port `_build_prompt`** (recipe → "A photographic, top-down shot of {name}[, a {cuisine} dish].
  plated on a wooden table, soft natural light, no text, no watermarks[. Visible ingredients: …]").
- **No key for the image provider** → a clear "add your {provider} image key" prompt (don't crash; backfill
  reports skipped/failed).
- **Save is DONE:** generated bytes → `RecipeRepository.setImage(recipeId, data, mime)` (CKAsset via
  RecipeImageCodec, cascade-parented to the recipe). Don't rebuild it.

## 3. Reuse / do-not-rebuild
- KeychainKeyStore + BYOKeyProvider's HTTPTransport + the AI-1 key UX (extend for the image key).
- RecipeRepository.setImage + RecipeImageCodec (CKAsset) — the save path, DONE.
- The image_provider setting (ProfileRepository, private plane) — read it.
- RecipeImageBackfillResult (generated/skipped/failed) — keep the shape.

## 4. Verification
- **Headless:** the image prompt builder (recipe → expected prompt string, empties collapsed); the
  OpenAI + Gemini image REQUEST bodies (mock transport — correct endpoint/body); response parsing
  (b64_json / inlineData → Data); the transient-vs-permanent error mapping + the failover decision.
  (Don't call the real image API.)
- **On-device (TestFlight):** with an OpenAI key + image_provider=openai → "Regenerate image" produces a
  header image saved as a CKAsset (appears on the recipe, syncs); backfill fills missing images
  (generated/skipped/failed reported); no-key → a clear prompt, not a crash.

## 5. Risks
- **Image key vs text key** — images need the image-provider's key (OpenAI reused; Gemini separate). The
  UX must make clear which key is needed; failover needs a Gemini key to engage.
- **gpt-image-1 access** — some OpenAI keys need org verification for gpt-image-1; the failover to Gemini
  (if keyed) + a clear error mitigate. (Don't hard-fail the whole backfill on one image's error.)
- **Large image bytes** — 1024×1024 PNG (~1-2MB) → CKAsset (RecipeImageCodec stages to Caches; fine).
- **Backfill cost** — many recipes × an image each = real API cost; backfill is user-triggered + reports
  counts. Maybe cap/confirm for a large library (flag in the UI).
- **No new CloudKit types** (RecipeImage type exists from SP-A Phase 3) → no schema deploy.
