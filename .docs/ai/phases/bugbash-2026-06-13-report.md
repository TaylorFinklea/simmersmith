# SimmerSmith — Ultracode Bug Bash & Architecture Review

**Date:** 2026-06-13  
**Method:** Multi-agent workflow — 23 subsystem bug-finders + 11 architecture-review agents, every bug finding adversarially verified (critical/high got a 3-lens refute/reproduce/independent panel, majority vote). 149 agents, ~9.2M tokens.  
**Result:** 55 confirmed bugs (20 low, 19 high, 16 medium), 10 findings refuted/dropped by verification, 11 architecture dimensions (62 findings).

> Findings only — no code changed. This is the audit record; fixes are tracked separately.

## Cross-cutting themes

The findings cluster into a handful of root causes. Fixing the cluster is higher-leverage than fixing items one by one.

### T1 — Post-M21 household migration debt (the dominant theme)
`user_id` is used where `household_id` belongs, and several UNIQUE constraints / PKs are still keyed on `user_id` even though the app became household-scoped at M21. This single root cause produces correctness bugs, silent data loss, duplicate rows, and a class of cross-member failures.
  
_Maps to:_ Bugs #8, #14, #16, #21, #23, #24, #34, #35, #49 · Arch: data-model 'uniqueness keyed on user_id', multi-tenancy 'user_id passed where household_id expected', entitlements 'keyed on user_id'

### T2 — Destroy-before-confirm / partial-write data loss
Mutating flows delete or partially write rows *before* a fallible step (AI call, validation) and then let the surrounding transaction commit the half-done state on failure.
  
_Maps to:_ Bugs #2 (rebalance_day deletes meals before the AI call), #19 (import_pricing commits a partial write on failure), #6 (resolve never commits a minted row), #50 (apply_ai_draft orphans sides)

### T3 — IDOR / cross-household reads via client-supplied ids
Endpoints fetch a row by a client-supplied id (`session.get(...)`) and mutate/return it without checking it belongs to the caller's household.
  
_Maps to:_ Bugs #5 (subscription /verify — account takeover), #13 (preference upsert), #17 (feedback upsert), #16 (feedback aggregation has NO scoping) · Arch: resolver reads cross-household private catalog; apply_ai_draft trusts raw recipe_id

### T4 — Event↔week grocery merge lifecycle
The events subsystem was the single buggiest slice (5 confirmed). The merge marker is the mutable event name and the lifecycle (delete / re-date / manual-merge / rename) is not reconciled against the week it merged into.
  
_Maps to:_ Bugs #9, #10, #11, #37, #38

### T5 — Freemium is declared but not enforced
Two gated actions and the entire conversational assistant run uncapped; the Apple webhook can leave a user Pro after expiry/refund; the paywall has no end-to-end test.
  
_Maps to:_ Bug #4 · Arch: entitlements 'two gated actions never enforced', 'assistant turns ungated → unbounded spend', testing CRITICAL 'paywall has no e2e test' · iOS #46/#47

### T6 — Outright crashes / broken features
A few defects take a whole feature down or corrupt output deterministically.
  
_Maps to:_ Bugs #18 (rebalance-day endpoint is a hard 500 — feature entirely non-functional), #3 (cancelled assistant turn makes the thread un-loadable / 500), #33 (recipe import 500s on '1/0'), #1 (kid-friendly preset corrupts every step & ingredient name)

### T7 — Operability blind spots
No logging configuration at all (info/debug dropped on the floor), no global exception handler, and provider URLs/error bodies leak into user-facing responses.
  
_Maps to:_ Arch: error-handling 'no logging config', 'no global exception handler', 'raw exception strings leak provider endpoints'

### T8 — OAuth/MCP hardening gaps
Scopes are advertised and threaded everywhere but enforced nowhere (every token is omnipotent); public/issuer URLs are derived from unauthenticated X-Forwarded-* headers with no TrustedHost pin; pending authorize rows never GC and 30-day bearers can't be revoked.
  
_Maps to:_ Arch: mcp-oauth all 6 findings

### T9 — iOS error-swallowing & locale fragility
A recurring iOS pattern: failures are caught into dead `@State` that's never rendered, or input is cleared before the server confirms; plus locale-dependent weekday/decimal handling that breaks non-US devices.
  
_Maps to:_ Bugs #15, #25, #27, #28, #29, #43, #44 · Arch: iOS 'no centralized 401 handling', 'optimistic mutations have no rollback'

**#1. Kid-friendly variation preset corrupts every step and ingredient name (missing tuple comma)**  
`app/services/recipe_ai.py:115` · data-loss · slice `ai-services` · votes 3/3

- **What:** The `kid_friendly` VariationPreset declares an ingredient rule `VariationRule(("onion"), "finely diced onion", ...)`. `("onion")` is a parenthesized string, NOT a single-element tuple (no trailing comma). The dataclass field `terms` therefore equals the string `"onion"`, and every place that iterates `rule.terms` iterates the *characters* `'o','n','i','o','n'` instead of the word.

Two consequences:
1. `_transform_ingredient` (line 613) gates on `any(term in name.lower() for term in rule.terms)` -> matches any ingredient name containing the letter o, n, or i (e.g. 'chicken broth', 'tomato', 'onion', 'cilantro', almost everything).
2. `_replace_term` (lines 601-602) runs `re.sub(re.escape(term), 'finely diced onion', ...)` once per character, with case-insensitive global replacement. Because the replacement string itself contains the matched characters, successive substitutions feed on their own output, producing exponential blow-up. `_transform_step` (lines 634-637) calls `_replace_term` for EVERY rule unconditionally (no match gate), so every step instruction of a kid-friendly variation is mangled.
- **Trigger:** Concrete trigger: `POST /api/recipes/{id}/variation-draft` with goal 'kid-friendly' (or 'kids'), or the suggestion route `POST /api/recipes/suggestion-draft` with goal 'kid' / 'family' (which calls build_variation_draft(anchor, goal='Kid-Friendly') via the kid_friendly_dinner preset). resolve_variation_preset normalizes to 'kid friendly' -> kid_friendly preset. I reproduced it: a step 'Combine the tomatoes and onions in a hot skillet.' becomes a multi-thousand-character garbage string of repeated 'finely diced onion' fragments, and 'chicken broth' becomes a similarly corrupted name. The user receives an unusable, garbled recipe draft.
- **Fix:** Change line 115 to `VariationRule(("onion",), "finely diced onion", "Use smaller onion pieces for a softer texture."),` (add the trailing comma so `terms` is a single-element tuple). Consider a defensive assertion that all VariationRule.terms are tuples to prevent recurrence.

**#2. rebalance_day deletes the day's meals before the AI call; AI failure commits the deletion as permanent data loss** _(finder said critical)_  
`app/services/assistant_tools.py:556-573` · data-loss · slice `assistant` · votes 3/3

- **What:** _run_rebalance_day deletes all WeekMeal rows (and their WeekMealIngredient rows) for target_date and calls session.flush() at lines 556-561 BEFORE invoking run_rebalance(). If run_rebalance() raises RuntimeError (AI provider 4xx/5xx/timeout — the documented failure mode, see the matching except at line 572), the function returns AssistantToolResult(ok=False) by a normal return. Because it returned rather than raised, the crash-rollback in run_tool (assistant_tools.py:888-898) never fires, and the tool_runner's session_scope (app/api/assistant.py:309) COMMITS the deletion. The user's meals for that day are permanently gone while the model/user are told 'Rebalance failed', implying nothing changed.
- **Trigger:** Trigger: user (or the LLM) calls rebalance_day for a day that has meals, and the rebalance AI call fails (provider timeout/429/5xx — common). Sequence: meals deleted + flushed -> run_rebalance raises RuntimeError -> return ok=False (no raise) -> run_tool skips rollback -> session_scope commits -> day's meals are gone. Contrast generate_week_plan which deletes only AFTER a successful AI response. Wrong outcome: silent permanent loss of a day's planned meals on a recoverable AI error.
- **Fix:** Call run_rebalance() FIRST and only delete existing meals after it succeeds, or wrap the delete+regenerate so the RuntimeError branch calls session.rollback() (or re-raises so run_tool's rollback fires) before returning ok=False.

**#3. Cancelled assistant turn persists status='cancelled', which is not a valid AssistantMessageOut status — thread becomes unreadable (500)**  
`app/api/assistant.py:417-433` · crash · slice `assistant` · votes 3/3

- **What:** When a turn is cancelled (client disconnect sets abort_event), respond_route sets final_status = 'cancelled' (line 417) and calls update_assistant_message(..., status=final_status) at line 426-429, committed by the session_scope at line 419. The AssistantMessage.status column is a free String(20) so 'cancelled' persists fine. But AssistantMessageOut.status (app/schemas/assistant.py:61) is Literal['queued','streaming','completed','failed'] — 'cancelled' is not allowed. assistant_thread_payload (app/services/presenters.py:512) emits every message into AssistantThreadOut, and GET /threads/{thread_id} (response_model=AssistantThreadOut) validates them.
- **Trigger:** Trigger: user starts a planning turn then closes the assistant sheet / loses connection mid-turn, so the disconnect watcher fires abort_event and run_assistant_turn returns cancelled=True. The cancelled status is committed. Next time the user (or list/detail) opens that thread via GET /api/assistant/threads/{thread_id}, FastAPI validates the AssistantThreadOut response, the message with status='cancelled' fails the Literal, and the request 500s. The thread is permanently un-loadable.
- **Fix:** Either add 'cancelled' to the AssistantMessageOut.status Literal (and the model default set), or map cancelled turns to an allowed status (e.g. 'completed' or 'failed') when persisting at line 417/429.

**#4. Apple webhook advances last_transaction_id on non-state-changing notifications, causing a later EXPIRED/REFUND with the same transactionId to be dropped (user keeps Pro after expiry/refund)**  
`app/api/subscriptions.py:201-216` · logic · slice `billing-entitlements` · votes 3/3

- **What:** In the webhook handler, when a notification is processed (not stale), `existing.last_transaction_id = verified.transaction_id` is set unconditionally (line 216), outside the `new_status` conditional. Unmapped notification types (PRICE_INCREASE, DID_CHANGE_RENEWAL_PREF, DID_CHANGE_RENEWAL_STATUS, etc.) yield `new_status is None` and intentionally do NOT change status/window — but they still bump `last_transaction_id`. The staleness check `transaction_is_stale` uses `int(incoming) <= int(stored)`, so any later notification carrying the SAME transactionId is then treated as stale and silently dropped.
- **Trigger:** Apple's App Store Server Notifications v2 embed the subscription's current/last transaction in `signedTransactionInfo`; within a single billing period EXPIRED, REFUND, DID_CHANGE_RENEWAL_STATUS, PRICE_INCREASE all carry the SAME transactionId (a new id only appears on renewal/resubscribe). Concrete sequence: (1) a DID_CHANGE_RENEWAL_STATUS or PRICE_INCREASE notification arrives with transactionId 200 -> new_status None -> status/window untouched but last_transaction_id is set to 200; (2) the subsequent EXPIRED or REFUND notification arrives with the same transactionId 200 -> transaction_is_stale(200, 200) returns 200<=200 == True -> the notification is dropped. The subscription status is never moved to expired/refunded, so is_pro() keeps returning True and the user retains Pro entitlement after cancellation/refund. The existing test test_unmapped_notification_type_does_not_advance_window already drives last_transaction_id from 100 to 200 via PRICE_INCREASE but never asserts last_transaction_id, so the regression is uncaught.
- **Fix:** Only advance last_transaction_id when the notification actually changes state (i.e. when new_status is not None), or move line 216 inside the `if new_status is not None:` branch. Alternatively, do not treat an equal transactionId as stale for terminal/status-changing notifications. Guard last_transaction_id advancement the same way the period window is guarded.

**#5. /verify migrates a subscription row to the caller with no ownership check (account takeover / IDOR when appAccountToken is unset)**  
`app/api/subscriptions.py:73-100` · security · slice `billing-entitlements` · votes 2/3

- **What:** verify_client_transaction looks up the existing Subscription by `apple_original_transaction_id` but never compares `existing.user_id` to `current_user.id`. The only guard is app_account_token_conflict, which requires BOTH tokens to be present. The iOS client (SubscriptionStore.purchase / product.purchase()) never passes a PurchaseOption.appAccountToken, so verified.app_account_token is always None and app_account_token_conflict always returns False. If the incoming transaction is not stale, upsert_subscription_from_transaction reassigns the row to the caller (services/subscriptions.py line 223: `row.user_id = user_id`).
- **Trigger:** Concrete sequence: User B obtains User A's signed JWSTransaction (e.g. via Family Sharing, a shared/jailbroken device, or a leaked receipt) and POSTs it to /api/subscriptions/verify with B's own bearer token. existing = A's row; app_account_token_conflict(None, None) -> False; if the transactionId is newer than A's last_transaction_id (e.g. after a renewal B's device synced), transaction_is_stale -> False, so upsert runs and sets row.user_id = B. A loses Pro and B gains it on A's dime. Even when stale, B receives A's full subscription details (_subscription_out of A's row) in the 200 response, leaking another account's billing state. The appAccountToken defense is documented as 'inert until iOS sets appAccountToken', and the iOS client demonstrably does not set it, so the protection is currently non-functional.
- **Fix:** Reject (409) or treat as a conflict when existing.user_id != current_user.id and the incoming appAccountToken does not positively bind the receipt to current_user; do not silently migrate the row to the caller on the verify path. At minimum require an appAccountToken match before re-binding, and have the iOS client set PurchaseOption.appAccountToken to the user id.

**#6. POST /api/ingredients/resolve mints a base ingredient but never commits, so the returned id points at a rolled-back row**  
`app/api/ingredients.py:479-497` · data-loss · slice `catalog` · votes 3/3

- **What:** resolve_ingredient_route calls resolve_ingredient(..., household_id=...) with the default persist=True. For a novel ingredient name that has no matching catalog row, resolve_ingredient mints a new BaseIngredient (and possibly an IngredientVariation) via ensure_base_ingredient/create_or_update_variation, which only session.flush() — they never commit. The route returns the resolution payload (including base_ingredient_id and resolution_status='resolved') but, unlike every sibling mutating route in this file (create/archive/merge/preference all call session.commit()), it omits session.commit(). The HTTP dependency get_session (app/db.py:48-53) only closes the session in its finally block — it does NOT commit — so the flushed INSERT is discarded on session.close().
- **Trigger:** Trigger: an authenticated client POSTs /api/ingredients/resolve with ingredient_name='dragon fruit puree' (a name not in the catalog). The response comes back with a non-null base_ingredient_id and resolution_status='resolved'/'unresolved-then-minted'. Because the route never commits and get_session never commits, the newly minted BaseIngredient row is rolled back when the request session closes. The client now holds a base_ingredient_id that references a row which does not exist in the database; a subsequent GET /api/ingredients/{that_id} returns 404, and any recipe/grocery row that stored that id is dangling. The MCP path (app/mcp/ingredients.py:95 via session_scope) commits and is unaffected, masking the bug; existing tests only resolve against pre-committed bases so they never exercise the mint path.
- **Fix:** Add `session.commit()` before returning in resolve_ingredient_route (matching the other mutating routes), or pass persist=False here and only persist on an explicit save flow.

**#7. merge_base_ingredients only migrates ONE preference row, orphaning all others onto the archived source base**  
`app/services/ingredient_catalog/variation.py:462-490` · data-loss · slice `catalog` · votes 3/3

- **What:** When merging two base ingredients, the code fetches preferences for the source with session.scalar(select(IngredientPreference).where(base_ingredient_id == source.id)) — a SINGLE row. It migrates/merges only that one row, then archives the source (source.active=False, source.archived_at set). But IngredientPreference is unique on (user_id, base_ingredient_id, rank), so a single base can have many preference rows: multiple users, AND multiple ranks per user (the whole M24 ranked primary/secondary feature). The companion merge_variations (lines 573-577) correctly iterates with .scalars(...).all(); the base merge does not.
- **Trigger:** Trigger: User A has rank=1 and rank=2 preferences on base 'ketchup-dup' (source), and/or User B also has a preference on it. Admin/household calls POST /api/ingredients/{source}/merge with target='ketchup'. merge_base_ingredients picks exactly one preference row, reassigns/merges it, then archives the source. Every remaining preference row still has base_ingredient_id = source.id, which is now an inactive/archived base. Those preferences are silently orphaned: choice_for_base_ingredient and the preference list will never surface them again, and they reference a dead base. The user's ranked brand list is lost. The route commits (app/api/ingredients.py:366-369) so the corruption is persisted.
- **Fix:** Iterate over ALL source preferences with .scalars(...).all() (like merge_variations does). For each, look up an existing (user, target, rank) row: if none, reassign base_ingredient_id=target.id; if a collision exists, merge the non-empty fields then delete the source pref — guarding against the (user_id, base_ingredient_id, rank) unique constraint.

**#8. staples UNIQUE is (user_id, normalized_name) but pantry dedup is household-wide — TOCTOU lets two members create duplicate staples**  
`app/models/profile.py:29` · concurrency · slice `data-model` · votes 3/3

- **What:** Staple.__table_args__ declares UniqueConstraint("user_id", "normalized_name", name="uq_staples_user_normalized_name"). The pantry service (app/services/pantry.py:112-122) explicitly dedupes household-wide — its own comment says per-user dedup is wrong "otherwise two members each adding 'Milk' both pass and the recurring auto-add then fires twice (M34)". But it enforces that invariant only with an application-level SELECT-then-INSERT; the backing DB constraint is still (user_id, normalized_name), which does NOT enforce household-wide uniqueness. update_pantry_item (pantry.py:156-161) likewise recomputes normalized_name on rename with no collision check, relying on a constraint that is on the wrong columns.
- **Trigger:** Concrete sequence: members A and B of household H both add a pantry item 'Milk' concurrently (two devices, or A on web + B on iOS). create_pantry_item runs the existence SELECT (pantry.py:115-120) for each; both see no existing household row and both INSERT. Because the unique constraint keys on user_id (A != B), both inserts succeed. The household now holds two 'Milk' staples, and apply_pantry_recurrings folds the recurring restock in twice — the exact double-add the code comment claims to prevent. The app-level check is a pure TOCTOU race with no DB backstop.
- **Fix:** Migrate uq_staples_user_normalized_name to UNIQUE(household_id, normalized_name) and update the model so the DB enforces the household-wide invariant the service relies on; add a collision check in update_pantry_item on rename. Dedupe existing per-household duplicates before adding the constraint.

**#9. Deleting a merged event orphans its grocery contribution on the week (stale event_quantity + zombie rows)**  
`app/services/events.py:216-217` · data-loss · slice `events` · votes 3/3

- **What:** delete_event() simply does session.delete(event). The event's EventGroceryItems cascade-delete (events.id FK is CASCADE), but the week-side GroceryItem rows that received this event's contribution are never reconciled. merge_event_into_week added each event row's quantity into GroceryItem.event_quantity (and created event-only GroceryItems with source_meals='event:<name>'). - **Trigger:** Create an event with a meal, let it auto-merge into a week (week GroceryItem.event_quantity becomes e.g. 5.0, plus a new 'event:Party' row for unmatched ingredients). DELETE /api/events/{id}. The event and its EventGroceryItems vanish, but the week's grocery list still shows the inflated event_quantity on matched rows and the orphaned 'event:Party' row forever — there is no longer any event to unmerge it. The shopping list is permanently wrong.
- **Fix:** In delete_event_route (or delete_event), if event.linked_week_id is set, resolve that week (household-scoped) and call unmerge_event_from_week(session, event=event, week=week) BEFORE session.delete(event).

**#10. Changing an already-merged event's date never re-points the grocery merge to the new week**  
`app/services/event_grocery.py:399-405` · correctness · slice `events` · votes 3/3

- **What:** _resolve_target_week short-circuits on event.linked_week_id and returns the previously linked week without re-checking event_date. update_event_route forwards a changed event_date and then calls apply_auto_merge_policy, but for an event that is already merged (linked_week_id set) the policy resolves to the OLD week via _resolve_target_week and merges there again. No code unmerges the old week or relinks to the week covering the new date (the update path, unlike the regenerate path, never clears linked_week_id).
- **Trigger:** Event dated 2026-06-15, auto_merge on, merged into the June 14-20 week (linked_week_id set). The host buys party groceries on the wrong week.
- **Fix:** When event_date changes and the event is already linked, unmerge from the current linked_week (clearing linked_week_id) before apply_auto_merge_policy, so _resolve_target_week falls through to the date-based lookup; or make _resolve_target_week prefer the date-covering week when event_date no longer falls inside linked_week's range.

**#11. Editing a manually-merged potluck event (auto_merge_grocery=False) silently drops the manual merge**  
`app/services/event_grocery.py:444-454` · data-loss · slice `events` · votes 3/3

- **What:** The manual merge endpoint POST /grocery/merge calls merge_event_into_week directly, which sets event.linked_week_id but does NOT set auto_merge_grocery. apply_auto_merge_policy treats auto_merge_grocery=False + linked_week_id set as 'should be unmerged' and calls unmerge_event_from_week. Every supplement add/edit/delete, meal add/edit/delete, and grocery refresh route runs regenerate_event_grocery + apply_auto_merge_policy, so the first such mutation after a manual merge force-unmerges the event from the week the user explicitly chose.
- **Trigger:** Potluck event with auto_merge_grocery=False. User POSTs /api/events/{id}/grocery/merge?week_id=W to fold their dishes into week W (week rows now include the event_quantity). User then adds a pantry supplement (or edits any meal). _refresh_event_after_supplement_change -> apply_auto_merge_policy sees auto_merge False + linked_week_id=W and unmerges, wiping the event_quantity contribution from week W. The deliberately-merged groceries silently disappear from the week.
- **Fix:** Either set event.auto_merge_grocery=True (or a separate 'manually_merged' flag) when the manual merge endpoint links a week, or make apply_auto_merge_policy only auto-unmerge when the link was established by the auto-merge policy rather than an explicit manual merge.

**#12. APNs 410 detection compares wrong field — dead device tokens are never disabled**  
`app/services/push_apns.py:159-166` · error-handling · slice `exports-push` · votes 3/3

- **What:** send_push checks `result.status == "Unregistered"` to detect an APNs 410 (device unregistered) and soft-disable the row. But the aioapns NotificationResult.status field holds the HTTP status CODE string ("200", "410", ...), not the reason. The reason string ("Unregistered") lives in result.description. For a 410, the real library sets result.status == "410" (see aioapns/common.py APNS_RESPONSE_CODE.GONE = "410" and connection.py on_data_received which sets status from the :status header and description=reason). So the elif branch is never taken; the code falls into the generic else and only logs `status=410` as a warning. disabled_at is never set.
- **Trigger:** Concrete trigger: a user deletes the app or otherwise invalidates their token. APNs replies 410 Unregistered. send_push receives result.status == "410" (not "Unregistered"), so the soft-disable branch at line 159 is skipped and device.disabled_at stays None. The dead row is selected again on every scheduler tick forever, send_push keeps returning it via _active_user_ids, and the explicit M38 pruning behavior documented in app/api/push.py:150 ("send_push prunes dead (410) device rows") never happens. The existing test tests/test_push.py:242 sets mock_result.status = "Unregistered", encoding the same wrong contract, so it passes while production is broken.
- **Fix:** Detect 410 by HTTP code, not reason: `elif getattr(result, "status", "") == "410" or getattr(result, "description", "") == "Unregistered":`. Update the mock in tests/test_push.py to set status="410", description="Unregistered" to match the real aioapns contract.

**#13. IDOR: preference upsert overwrites another user's signal by preference_id with no ownership check**  
`app/services/preferences.py:46-69` · security · slice `household-multitenancy` · votes 3/3

- **What:** upsert_preference_signals fetches a PreferenceSignal by primary key from the client-supplied preference_id (session.get(PreferenceSignal, payload.preference_id)) without verifying signal.user_id == user_id. When the row exists it is used directly, skipping both the user-scoped fallback lookup and the new-row creation, and the mutation at lines 60-69 overwrites signal_type/name/normalized_name/score/weight/rationale/source/active/updated_at while never setting or checking user_id. The victim's user_id stays intact, so the row simply has its content rewritten by the attacker.
- **Trigger:** Authenticated user A calls POST /api/preferences with signals=[{preference_id: <B's signal id>, signal_type, name, score, active:false, ...}]. preference_id values are real UUIDs returned to clients in every PreferenceContextResponse (preference_signal_payload sets 'preference_id': signal.id), so they leak across the API surface. session.get returns B's row regardless of owner; the code mutates it and commits. A can silently flip another user's preference score or set active=False on a victim's avoid/allergy signal (which score_meal_candidate relies on to block disliked/allergen meals), corrupting B's data and weakening B's meal-safety filtering. Cross-tenant write with no authz.
- **Fix:** After session.get, discard the row if signal.user_id != user_id (treat as not found and fall through to the user-scoped lookup / create path). Or fetch with a combined filter: session.scalar(select(PreferenceSignal).where(PreferenceSignal.id == payload.preference_id, PreferenceSignal.user_id == user_id)).

**#14. update_profile deletes ALL household members' staples but re-inserts only the caller's, causing cross-member pantry data loss**  
`app/services/profile.py:26-44` · data-loss · slice `household-multitenancy` · votes 2/3

- **What:** update_profile runs delete(Staple).where(Staple.household_id == household_id) (household-wide) then re-inserts only the staples in the request payload with user_id=<caller>. Staples are household-scoped on every read/aggregation path (presenters.profile_payload, grocery.py, pantry.py all filter by Staple.household_id), and pantry.create dedups household-wide because 'the pantry is shared across the household'. The unique constraint is on (user_id, normalized_name), so members co-own household-scoped rows under different user_ids.
- **Trigger:** In a shared household X (created when member B claims member A's invite; claim_invitation -> merge_solo_into re-points B's staples to household_id=X while keeping user_id=B), suppose A has staple 'Milk' (user_id=A, household_id=X) and B has 'Bread' (user_id=B, household_id=X). Member B opens the app (GET /api/profile returns the merged household pantry) and saves any profile change. put_profile -> update_profile deletes WHERE household_id=X, removing BOTH 'Milk' and 'Bread', then re-inserts only what B's client sent, all stamped user_id=B. - **What:** The best-effort image generation shares the recipe-save transaction. persist_recipe_image INSERTs a RecipeImage row keyed by recipe_id (PK). The pre-check recipe_has_image() and the INSERT are not atomic, and a failing INSERT poisons the session so the later session.commit() at line 285 re-raises past the best-effort except blocks.
- **Trigger:** Two concurrent saves of the same recipe both observe recipe_has_image()==False (line 265), both call persist_recipe_image -> RecipeImage INSERT with the same primary key recipe_id. The second flush/commit raises IntegrityError. That exception either escapes the inner try (it surfaces at the commit on line 285, which is outside the try) or, if raised earlier inside the try, is swallowed by the broad except but leaves the session in a failed-transaction state so line 285's session.commit() still raises. Net effect: an image-layer conflict aborts a recipe save that should have succeeded, contradicting the documented 'must never block the save' intent.
- **Fix:** Run the opportunistic image write in a nested transaction/savepoint (session.begin_nested) or a separate session/commit so its failure can't poison or roll back the recipe save; on IntegrityError, treat the image as already-present and continue.

**#55. score_macro_drift always returns no flags — 'Days off calorie target' note is dead**  
`app/services/week_planner.py:527-536` · logic · slice `weeks-grocery` · votes 1/1

- **What:** score_macro_drift reads per-meal macros from `meal.get('ingredients')` on meal_plan entries. But generate_week_plan defaults each meal_plan entry's `ingredients` to `[]` (week_planner.py:600) and the AI prompt only places ingredients on the `recipes` array, never on meal_plan items (prompt structure lines 336-340). So calculate_meal_macros always receives `[]`, returns an empty MacroBreakdown (is_empty True, nutrition.py:495), and every meal is skipped, making score_macro_drift always return [].
- **Trigger:** The 'Days off calorie target: ...' annotation appended to week_notes in the generate endpoint (weeks.py:188-198) can never appear, because macro_flags is always empty. The whole macro-drift feedback feature produces no output for any generated plan. The function would need to resolve each meal_plan entry's recipe and pull that recipe's ingredients (or be fed recipe ingredients) to compute meaningful per-day calories.
- **Fix:** Resolve each meal_plan entry to its recipe in the `recipes` array (by recipe_name) and sum that recipe's ingredients, rather than reading the always-empty meal-level `ingredients`.

## Part 2 — Architecture review

### Data model integrity

_The relational core is competently built where it uses real foreign keys: the parent/child subtrees (Week→WeekMeal→ingredients/sides, Recipe→ingredients/steps, Event→meals/grocery, ExportRun→items, etc.) all carry proper FKs with deliberate, mostly-correct ondelete semantics (CASCADE down the ownership tree, SET NULL for soft references like recipe_id), and ORM relationship cascades are consistent with them. Indexes on hot scoping columns and self-referential merge/variant chains are present and sensible, and the migration history shows real care (every migration has a working downgrade, dialect branching is deliberate, the deferrable-unique and widen-FK fixes are well-reasoned). However, the dimension has two systemic weaknesses that dominate its risk profile. First, the multi-tenant boundary is not enforced at the schema level: household_id — the documented tenancy unit — is a bare String(36) on every content table with NO foreign key to households.id and NO FK to users.id (only image_gen_usage and the household_* membership tables reference their parents), so the entire tenant graph relies on application discipline for both referential integrity and cascade cleanup. Second, the absence of a metadata naming_convention forces nearly every schema-changing migration to fork into hand-written Postgres-vs-SQLite paths with full-column-list table rebuilds, an error-prone pattern that has already produced model/migration drift. Neither is an active outage, but both are foundational and get more expensive to fix with every migration added._

- **[HIGH] household_id is the tenancy unit but has no foreign key on any content table**
  - _Area:_ app/models/{recipe,week,event,profile,catalog,aliases}.py; alembic/versions/20260501_0027_households.py, 20260530_0043_household_id_not_null.py
  - _Observation:_ Households are the documented unit of shared planning, and weeks/recipes/staples/events/guests (plus base_ingredients.household_id, household_term_aliases, etc.) all carry household_id. Yet only household_members, household_invitations, and household_settings declare a ForeignKey to households.id (grep for 'households.id' returns exactly those three). On the content tables household_id is a plain String(36) — indexed, and as of migration 0043 NOT NULL, but with zero referential integrity. The 0027 migration that introduced the column added an index but deliberately no FK, and that was never revisited.
  - _Risk:_ Two failure modes. (1) Orphaning: deleting or mis-pointing a Household leaves every Week/Recipe/Staple/Event/Guest (and their whole FK subtrees) pointing at a non-existent household with no DB protection; such rows silently vanish from every `household_id == X` filter (exactly the drift 0043's own docstring warns about for NULLs). merge_solo_into in services/households.py manually re-points content before deleting the solo household — correct today, but the invariant lives entirely in that one code path; any future delete/merge that forgets it corrupts tenancy with no backstop. (2) No cascade: a legitimate household teardown can't rely on ON DELETE CASCADE, so cleanup must enumerate every scoped table by hand forever.
  - _Recommendation:_ Add real FKs household_id -> households.id on the content tables. Choose semantics deliberately: ON DELETE RESTRICT (force explicit re-point/teardown, matching current merge logic) or ON DELETE CASCADE (let the DB tear down a household's content). Do it in one migration across the SHARED_TABLES set already enumerated in 0027/0043, mirroring the existing batch_alter_table-for-SQLite pattern. At minimum add RESTRICT so the orphaning class of bug becomes impossible at the schema level.
- **[HIGH] No metadata naming_convention forces hand-written dual-dialect migrations and has caused real drift**
  - _Area:_ app/db.py (Base = DeclarativeBase, no naming_convention); alembic/versions/20260410_0014_users_and_multi_tenant.py, 20260504_0031_household_ingredients.py, and most schema-change migrations
  - _Observation:_ Base is a bare DeclarativeBase with no MetaData(naming_convention=...). Implicit constraint/index names are therefore unpredictable across dialects and SQLAlchemy versions (profile_settings_pkey, sqlite_autoindex_base_ingredients_N, <table>_<column>_key). The consequence is visible throughout the migration history: 0014 forks into _upgrade_postgres/_upgrade_sqlite with full hand-written CREATE TABLE ... weeks_new column lists; 0031 introspects pg_constraint to discover the unique-constraint name and rebuilds the SQLite table via copy_from with a 27-column hand-transcribed Table; named UniqueConstraints in the models exist precisely to dodge this. Each such rebuild hardcodes the complete column set, so a column added before the rebuild that isn't transcribed would be silently dropped.
  - _Risk:_ High maintenance cost and latent data-loss risk on every future ALTER. The full-column-list rebuild pattern is the single most dangerous recurring construct here: an omission silently drops a column on SQLite (and the test suite, which runs SQLite, may not catch a column that only matters in prod). It also blocks Alembic autogenerate from being trustworthy, since generated names won't match the ad-hoc ones already in the DB.
  - _Recommendation:_ Adopt a standard naming_convention on Base.metadata (the canonical ix/uq/ck/fk/pk template) and configure alembic env.py with render_as_batch=True for SQLite. New constraints then get deterministic names referenceable directly with op.drop_constraint on both dialects, collapsing the dual-path migrations. Note this won't rename constraints already created under the old implicit names, so pair it with a one-time reconciliation pass; the payoff is that every migration from here forward stops needing bespoke SQLite table rebuilds.
- **[MEDIUM] Tenant-scoped uniqueness and ownership are keyed on user_id, not household_id, contradicting the sharing model**
  - _Area:_ app/models/week.py (uq_weeks_user_week_start), profile.py (Staple.uq_staples_user_normalized_name, PreferenceSignal, DietaryGoal PK, ProfileSetting PK), catalog.py (IngredientPreference), billing.py (UsageCounter)
  - _Observation:_ Post-M21 the tenancy unit is the household, and Week/Staple/Event/Guest all carry household_id. But their uniqueness invariants and several primary keys are still keyed on user_id: Week is UNIQUE(user_id, week_start), Staple is UNIQUE(user_id, normalized_name), IngredientPreference/PreferenceSignal key on user_id, and DietaryGoal/ProfileSetting/Subscription are PK'd on user_id. In a multi-member household, two members can each create a Week for the same week_start in the same household, or two Staples with the same normalized_name — the constraint that should make a household's plan singular is scoped to the individual.
  - _Risk:_ Duplicate/conflicting household-shared rows. Two members planning the same week independently produce two Week rows for one (household, week_start); grocery aggregation, the iOS current-week fetch, and weeks_get_by_start can then pick an arbitrary one or show duplicates. Same for duplicate staples within a household. This is a correctness gap that only manifests in genuinely shared (multi-member) households, so it's invisible in the common solo case and in solo-household tests.
  - _Recommendation:_ Decide per table whether the invariant is per-user or per-household and align the constraint. Week and Staple are almost certainly per-household (UNIQUE(household_id, week_start) / UNIQUE(household_id, normalized_name)); preferences/goals may legitimately stay per-user. Migrate the constraints that should be household-scoped, de-duping existing rows first (as 0046 did for memberships). Until then, document the multi-member duplication as a known limitation.
- **[MEDIUM] ImageGenUsage model omits foreign keys that the migration declares (model/schema drift)**
  - _Area:_ app/models/image_usage.py vs alembic/versions/20260430_0026_image_gen_usage.py
  - _Observation:_ Migration 0026 creates image_gen_usage with FK user_id -> users.id ON DELETE CASCADE and recipe_id -> recipes.id ON DELETE SET NULL. The ImageGenUsage ORM model declares both columns as plain String with no ForeignKey and no relationship. So the live DB enforces referential integrity and cascade that the model is unaware of. Notably this is the ONLY table in the schema with a user_id -> users.id FK — every other user-scoped table (subscriptions, push_devices, ai_runs, assistant_threads, weeks, recipes, staples, preference_signals, ingredient_preferences, dietary_goals, profile_settings, usage_counters, guests, events) has user_id as a bare String with no FK.
  - _Risk:_ Two consequences. (1) Drift: the model misrepresents the schema; an Alembic autogenerate run against these models would try to DROP the two FKs, and a developer reading the model would wrongly conclude no integrity exists. (2) Inconsistent user-deletion behavior: image_gen_usage rows cascade-delete on user removal, but every other user-owned table orphans, so there is no coherent account-deletion story — partial cleanup at best. Combined with the household_id finding, there is effectively no enforced cleanup path for deleting a user or household.
  - _Recommendation:_ Make model and schema agree. Either add the two ForeignKeys to ImageGenUsage to match the DB, or (if the bare-String convention is intentional for user_id everywhere) drop the lone FK and treat image_gen_usage like the rest. Separately, decide on a deliberate, uniform user_id integrity/cleanup policy across all user-scoped tables rather than the current one-off.
- **[MEDIUM] Cross-dialect uniqueness divergence: DEFERRABLE-unique and partial-unique indexes are Postgres-only, leaving SQLite tests with weaker invariants**
  - _Area:_ app/models/week.py (uq_week_day_slot DEFERRABLE); alembic/versions/20260522_0042, 20260504_0031_household_ingredients.py
  - _Observation:_ Several integrity rules behave differently between the prod (Postgres) and test (SQLite) databases by design. uq_week_day_slot is DEFERRABLE INITIALLY DEFERRED on Postgres but immediate on SQLite (0042 early-returns on non-Postgres); the base_ingredients catalog uniqueness is two partial unique indexes (WHERE household_id IS [NOT] NULL) implemented with postgresql_where/sqlite_where. The model docstrings acknowledge this. The meal-swap path that motivated the deferral is, per the 0042 comment, exercised end-to-end only on Postgres.
  - _Risk:_ The test database enforces a materially different constraint surface than production. The immediate-vs-deferred difference means the exact concurrency/transaction interleaving that broke meal-swap (transient duplicate during a two-row UPDATE) cannot be reproduced or regression-tested in SQLite — a future change to the swap path could reintroduce the IntegrityError and pass CI. More broadly, any data-integrity logic depending on deferred checks or partial-unique semantics is unverified by the SQLite suite.
  - _Recommendation:_ Accept the dual-dialect reality but add Postgres-backed integration coverage for the constraints that diverge (meal swap against deferred unique; the two base_ingredients partial uniques). A lightweight dockerized-Postgres test job for just these paths closes the gap without converting the whole suite. Document in each affected model exactly which invariant is Postgres-only so future editors don't assume SQLite parity.
- **[LOW] Wide self-referential SET NULL chains and soft recipe references can silently break planning data on delete**
  - _Area:_ app/models/recipe.py (base_recipe_id, recipe_template_id), catalog.py (BaseIngredient.merged_into_id, IngredientVariation.merged_into_id), week.py / event.py (recipe_id SET NULL on WeekMeal, WeekMealSide, EventMeal, FeedbackEntry; merged_into_grocery_item_id)
  - _Observation:_ The schema makes heavy, consistent use of ON DELETE SET NULL for soft references: WeekMeal.recipe_id, WeekMealSide.recipe_id, EventMeal.recipe_id, AssistantMessage.attached_recipe_id, base/variation merged_into_id, EventGroceryItem.merged_into_grocery_item_id, etc. This is the right default (deleting a recipe shouldn't delete the week that referenced it), but it means a recipe delete or an ingredient merge can null out references across many tables, and the remaining denormalized columns (recipe_name, ingredient_name, normalized_name copied onto the meal/grocery rows) become the only surviving link.
  - _Risk:_ Mostly correct and intentional, but two soft spots: (1) self-referential SET NULL on merged_into_id chains means a merge target deletion silently breaks the merge lineage (merged_into becomes NULL, the row looks un-merged); (2) rows whose recipe_id was nulled rely entirely on the denormalized name columns, so any code path that assumes a non-null recipe_id when source/ai_generated flags suggest a real recipe could mis-handle the orphaned-but-named row. These are edge cases, not systemic, hence low severity.
  - _Recommendation:_ No structural change needed — SET NULL is the right call. Add a couple of invariant tests: deleting a Recipe nulls WeekMeal/EventMeal/Side.recipe_id while preserving recipe_name and leaving the meal queryable; merging then deleting a BaseIngredient leaves dependent rows resolvable via normalized_name. This locks in the denormalization-as-fallback contract that the SET NULL design implicitly depends on.

### Multi-tenancy & isolation

_Multi-tenancy is, overall, in strong shape and clearly engineered with intent. The foundation is excellent: every authenticated principal carries a resolved household_id (CurrentUser), domain reads/writes go through household-scoped loaders (get_recipe/get_week/get_event/load_*_or_404 all filter on household_id), the MCP server reuses the exact same FastAPI route handlers so there is a single enforcement point per operation rather than a parallel under-guarded surface, and the catalog has a deliberate, tested global-approved vs household-private tier model with dedicated IDOR guards (_require_visible_/_require_owned_) on its routes plus a regression suite (test_ingredient_idor, test_catalog_scoping, test_isolation). The one genuine systemic gap is the core ingredient resolver: resolve_ingredient and its normalized-name lookups (_active_base_by_normalized_name and the variation-by-name query) resolve and link base/variation rows with NO household visibility filter. The team patched the nutrition-calorie symptom of this (M8 _scope_nutrition_refs) but not the resolver itself, so another household's private base/variation name can still be read and its id persisted onto a recipe/meal ingredient. A secondary latent hazard is user_id/household_id argument conflation in gather_planning_context, which currently fails closed (empty results) but is the exact shape of bug that becomes a leak if it ever lands in a returning getter. None of the standard CRUD surfaces show an exploitable IDOR._

- **[HIGH] Ingredient resolver links and reads cross-household private catalog rows (resolver bypasses the visibility tier)**
  - _Area:_ app/services/ingredient_catalog/__init__.py (resolve_ingredient), app/services/ingredient_catalog/shared.py (_active_base_by_normalized_name, _active_variation_by_normalized_name); reachable via app/api/recipes.py (save/import/estimate), app/api/ingredients.py resolve_ingredient_route, app/services/drafts.py apply_ai_draft, and all MCP recipe/ingredient tools
  - _Observation:_ resolve_ingredient resolves base/variation references with NO household/submission-status filter on three paths: (1) caller-supplied base_ingredient_id / ingredient_variation_id are loaded via session.get() and only checked for active/not-archived (lines 103-111), never for visibility; (2) variation-by-normalized-name (lines 113-123) and (3) base-by-normalized-name via _active_base_by_normalized_name (shared.py line 262) run plain `WHERE normalized_name == X` queries with no `submission_status == approved OR household_id == caller` clause. The 0031 partial-unique indexes explicitly allow a household's private `household_only` row and other households' private rows to coexist for the same normalized_name, so session.scalar() can return a foreign household's private row. The resolver then returns base_ingredient_name / ingredient_variation_name in its payload and the caller persists the foreign base_ingredient_id/ingredient_variation_id onto its own RecipeIngredient/WeekMealIngredient. The catalog ROUTES are correctly guarded (_require_visible_/_require_owned_), and the nutrition path was hardened (M8 _scope_nutrition_refs in recipes.py), but those defenses sit outside and after the resolver; resolve_ingredient is the shared chokepoint and is unguarded.
  - _Risk:_ Cross-tenant private-catalog read. Household B, by importing/saving a recipe or calling /api/ingredients/resolve (or the MCP ingredients_resolve tool) with an ingredient whose normalized name collides with — or by passing the id of — household A's private `household_only` base/variation, gets back A's private ingredient name (and, before the M8 scrub, its nutrition values) and silently links B's data to A's private row id. The existing test_nutrition_estimate_ignores_other_household_private_ref proves the team knows the class exists but only covers the nutrition-estimate symptom; the name-leak + foreign-id-link via the resolver itself is untested and unblocked. Also fragments the tier model (B's rows now reference A's private base that A can archive/merge).
  - _Recommendation:_ Push the household visibility rule INTO the resolver, not just the routes. Thread household_id into _active_base_by_normalized_name / the variation-by-name query and the explicit-id session.get() branches, and reject/ignore any row where submission_status != 'approved' and household_id != caller (mirror search_base_ingredients' `or_(approved, household match)` filter, which is already correct). Treat household_id as required for any caller-facing resolve; only the seed/system path (household_id=None) should be allowed to match approved-only. Add a regression test: A creates a private base with a unique normalized_name (no global equivalent), B resolves that name and must get unresolved/a fresh B-private row, never A's id or name.
- **[MEDIUM] resolve_ingredient is called with household_id=None on internal-but-data-bearing backfills, widening the unscoped surface**
  - _Area:_ app/services/ingredient_catalog/__init__.py: ensure_catalog_defaults() and the _backfill_rows helper call resolve_ingredient with no household_id; reresolve_unresolved_for_household passes household_id correctly
  - _Observation:_ ensure_catalog_defaults() and its _backfill_rows() resolve RecipeIngredient/WeekMealIngredient/GroceryItem rows across ALL households (it iterates select(RecipeIngredient) with no household filter) while calling resolve_ingredient without household_id. With household_id=None the resolver's new-row branch mints into the GLOBAL approved tier (new_base_status='approved') and its name lookups are global. Combined with finding 1, a system-wide backfill can both promote one household's ingredient names into the shared approved catalog and cross-link households' rows to whatever normalized-name match happens to exist first.
  - _Risk:_ A maintenance/seed run silently leaks private ingredient names into the global master list and cross-pollinates base_ingredient_id links between households. Lower blast radius than finding 1 because it is not a per-request attacker path, but it runs with the broadest scope and no household boundary, so a single invocation can corrupt the tier separation globally.
  - _Recommendation:_ Either restrict ensure_catalog_defaults/_backfill_rows to operate per-household (group rows by their owning recipe/week household_id and pass that household_id into resolve_ingredient), or have it resolve against approved-only rows and never mint household-private content as approved. Once finding 1's resolver guard exists, audit every resolve_ingredient call site for a missing household_id (grep shows these backfills and any future internal callers are the risky ones).
- **[MEDIUM] user_id passed where household_id is expected in gather_planning_context (fail-closed today, leak-shaped)**
  - _Area:_ app/services/week_planner.py gather_planning_context (lines 115, 117): staple_names(session, user_id) and list_weeks(session, user_id, limit=4)
  - _Observation:_ gather_planning_context takes a household_id keyword (used only for aliases at line 155) but calls the household-scoped readers with user_id: staple_names(session, household_id) and list_weeks(session, household_id) both filter on Staple.household_id / Week.household_id, yet receive user_id. Because solo households get a distinct new_id() household_id (user_id != household_id), these two queries match nothing in production — pantry and recent-meal planning context come back empty rather than cross-household. This is the same user_id/household_id conflation the codebase elsewhere is careful to avoid, and the comment at line 150-152 ('single-member households use user_id == household_id') encodes an assumption that is false in this schema.
  - _Risk:_ Today: a correctness/quality bug — week-generation prompts silently lose pantry-on-hand and recent-meal-variety context, degrading AI plans, with no cross-tenant exposure (it queries a non-existent id and returns empty). Latent: the moment this same arg-swap pattern appears in a getter that returns rows on a partial/looser match, or if household_id is ever made to equal user_id for legacy rows, it flips from fail-closed to a cross-boundary leak. It signals the household_id/user_id boundary is not consistently enforced by types.
  - _Recommendation:_ Pass household_id (not user_id) to staple_names and list_weeks in gather_planning_context, and require household_id as a positional param rather than an optional kwarg defaulting to user_id. Broadly, make household-scoped service functions name their parameter household_id and avoid any `household_id or user_id` fallbacks (also present at line 155) — these fallbacks are the systemic smell; a single mis-wire becomes invisible because both are opaque 36-char strings.
- **[LOW] IngredientPreference upsert accepts any base/variation id without a household-visibility check**
  - _Area:_ app/services/ingredient_catalog/variation.py upsert_ingredient_preference; exposed via app/api/ingredients.py upsert_ingredient_preference_route and MCP ingredient_preferences_upsert
  - _Observation:_ upsert_ingredient_preference validates only that base_ingredient_id exists and that preferred_variation_id (if given) belongs to that base — it never checks the base is visible to the caller's household (unlike the catalog read/mutate routes which gate on _require_visible_/_require_owned_). This closes the last unguarded id-acceptance point in the ingredient surface and makes the preference path consistent with the rest of the catalog API.
- **[LOW] apply_ai_draft / update_week_meals accept raw meal.recipe_id without household ownership re-check**
  - _Area:_ app/services/drafts.py apply_ai_draft (the `recipe = session.get(Recipe, recipe_id)` branch for meals whose recipe_id is not in known_recipes) and update_week_meals (meal.recipe_id assigned straight from the payload); app/services/recipes.py mark_week_recipe_usage (session.get(Recipe, meal.recipe_id))
  - _Observation:_ Unlike upsert_recipe (which scopes base_recipe_id to the household and refuses foreign ids) and the sides path (which validates recipe_id via _validate_recipe_id against household_id), the week-meal recipe_id is trusted as-is. apply_ai_draft does session.get(Recipe, recipe_id) with no household filter, and mark_week_recipe_usage likewise dereferences and writes last_used on whatever Recipe that id points to. Fix mark_week_recipe_usage to only touch recipes in the week's household. - **[INFO] Household-scoping invariant is enforced by convention per-query, with no structural backstop (no RLS / no shared scoped-session guard)**
  - _Area:_ Cross-cutting: app/api/*.py loaders, app/services/*.py, app/db.py (plain Session, no tenant context); models carry household_id but rely on every query adding the filter
  - _Observation:_ Isolation rests entirely on each query remembering to add `.where(Model.household_id == current_user.household_id)`. The pattern is applied impressively consistently (and the MCP-reuses-routes design collapses the surface to one enforcement point per op), but there is no defense-in-depth layer: no Postgres row-level security keyed on a session GUC, and no scoped-session/repository wrapper that injects the household filter. The single proven gap (the resolver) is exactly a spot where a developer reached for session.get()/a bare normalized-name query and the convention silently didn't apply.
  - _Risk:_ Future leaks are one forgotten WHERE clause away, and they will pass tests that don't specifically construct a two-household scenario. The existing two-household tests cover the catalog and a handful of CRUD surfaces but not every getter, so regressions in newer modules can ship unnoticed.
  - _Recommendation:_ Add a structural backstop sized to the team's appetite: (a) cheapest — a session-level SQLAlchemy event or a lint/CI grep that flags session.get(<HouseholdScopedModel>, ...) and bare selects on household-scoped tables outside the approved loader helpers; (b) stronger — Postgres RLS policies on recipes/weeks/events/staples/base_ingredients keyed on a per-request `SET app.household_id`, set in get_session from CurrentUser. Either turns 'every author must remember' into 'the platform enforces it', and would have caught finding 1.

### Concurrency & transactions

_This dimension is, on the whole, unusually well-considered for an app this size — the team clearly understands the hard parts. The two genuinely racy single-use flows are handled correctly and idiomatically: invite-claim serializes on a with_for_update row lock (no-op on SQLite tests, enforced on Postgres prod), and the monthly usage counter is a dialect-dispatched INSERT ... ON CONFLICT DO UPDATE ... RETURNING that is atomic against both duplicate-insert and lost-update races. First-sign-in user creation and solo-household creation both carry explicit IntegrityError recovery keyed on the right unique constraints (uq_household_members_user). Commit ownership is consistent within the REST layer (get_session never commits; every mutating route commits explicitly; services flush-only). The remaining risk is concentrated in three places: (1) the subscription verify/upsert path is a non-atomic read-modify-write across two unique columns with no IntegrityError recovery — the one place the codebase forgot to apply the pattern it uses everywhere else; (2) create_solo_household's recovery does a full session.rollback() that, when called mid-transaction from remove_member, silently reverts the caller's already-flushed delete and returns success-without-effect; and (3) the MCP surface composes session_scope (auto-commit) around REST handlers that self-commit, which is benign today but erodes the all-or-nothing guarantee and leaves a latent nested-scope hazard if an events-style dual-session route is ever exposed via MCP. None of these are systemic; they're a short, specific list against an otherwise disciplined baseline._

- **[HIGH] Subscription upsert is a non-atomic read-modify-write across two unique columns with no IntegrityError recovery**
  - _Area:_ app/services/subscriptions.py:182-237 (upsert_subscription_from_transaction); app/api/subscriptions.py:70-100 (client verify route); app/models/billing.py:12-27 (Subscription: user_id PK, apple_original_transaction_id unique)
  - _Observation:_ upsert_subscription_from_transaction does SELECT-by-original_transaction_id, then SELECT-by-user_id, then either INSERT or mutate-in-place, then flush — with no with_for_update and no on_conflict. The Subscription table has TWO unique constraints in play: user_id is the primary key and apple_original_transaction_id is unique. The client verify route (app/api/subscriptions.py:92-99) calls this then commits, with no try/except IntegrityError around it. This is the exact pattern the codebase handles correctly elsewhere (auth.py first-sign-in, households.create_solo_household both catch IntegrityError and re-read), but here the recovery is simply absent.
  - _Risk:_ Two concurrent client receipt-validation requests for the same user (double-tap on Restore Purchases, an automatic retry racing the original, or app-foreground re-validation racing a manual tap) both SELECT None and both attempt INSERT; the second commit raises IntegrityError on the user_id PK or the apple_original_transaction_id unique index, which propagates as an unhandled 500. The same 500 occurs if a webhook-triggered create (future) races a client create for the same original transaction. Worse than a transient error: a 500 on the purchase-confirmation path can leave the client believing entitlement failed and prompt the user to re-buy or open a support ticket, while the row may or may not exist depending on which writer won.
  - _Recommendation:_ Make this path match the rest of the codebase. Either (a) wrap the upsert+commit in try/except IntegrityError that rolls back and re-reads the row by apple_original_transaction_id (mirroring auth.py:74-85 and households.py:131-139), returning the surviving row; or (b) convert to a dialect-dispatched INSERT ... ON CONFLICT (apple_original_transaction_id) DO UPDATE ... RETURNING exactly like entitlements.increment_usage already does. Option (b) is the more robust fit since the increment path proves the team has the helper pattern. Add a regression test that fires two concurrent verifies for one original_transaction_id and asserts a single row and two 200s.
- **[HIGH] create_solo_household's full-session rollback silently reverts a caller's already-flushed work (remove_member returns 204 with no effect)**
  - _Area:_ app/services/households.py:131-140 (create_solo_household IntegrityError branch) and households.py:472-503 (remove_member, which deletes+flushes then calls create_solo_household); commit in app/api/household.py:286
  - _Observation:_ create_solo_household recovers from a concurrent-creation IntegrityError with a bare session.rollback() (line 138) — which rolls back the ENTIRE transaction, not just the failed member insert. This is correct when create_solo_household is the only work in the unit (its two other callers, _resolve_or_create_household_id and seed_defaults, start fresh). But remove_member first does session.delete(target) + session.flush() (lines 498-499), THEN calls create_solo_household(session, target_user_id) (line 503) inside the same transaction. If that inner insert hits the uq_household_members_user race (the removed user signed in on another device and already got a solo created), the inner rollback() discards the outer member deletion too. remove_member then returns the existing household_id, the route commits an empty transaction (app/api/household.py:286), and responds 204 Created/No Content — but the member was never actually removed.
  - _Risk:_ A kicked/left member silently remains in the household under a specific concurrency window (removed user's device re-authenticating concurrently). The API reports success, so neither the operator nor the client retries. Multi-tenant blast radius: the supposedly-removed user retains read/write access to shared household data (recipes, weeks, events) they should have lost. Low frequency but a real authorization-boundary failure, not just a UX glitch.
  - _Recommendation:_ Decouple the recovery from the caller's transaction. Give create_solo_household an IntegrityError path that uses a SAVEPOINT (session.begin_nested()) around just the member insert so only that insert unwinds, leaving the caller's prior delete intact; or restructure remove_member so the solo creation cannot partially-revert the removal (e.g. perform the removal and the new-solo creation as two separately-committed steps, or assert post-conditions before the route commits). Add a test: concurrent remove_member + sign-in for the same target user, assert the member row is gone after commit.
- **[MEDIUM] MCP composes session_scope (auto-commit) around REST handlers that self-commit, eroding the unit-of-work guarantee**
  - _Area:_ app/mcp/weeks.py, recipes.py, ingredients.py, profile.py (each tool: `with session_scope() as session: _call_route(lambda: <route_handler>(..., session=session))`); REST handlers commit internally e.g. app/api/weeks.py:127/141/206-208, app/api/recipes.py multiple
  - _Observation:_ Every MCP tool opens a session_scope (which commits on clean exit) and inside it invokes a FastAPI route handler that ALSO calls session.commit() one or more times on that same session. So the route's internal commit IS the durable boundary; session_scope's trailing commit is redundant. This means session_scope's headline guarantee — 'commit on success, roll back the whole unit on exception' — does not actually hold for MCP tool calls: a route that commits at step A and then raises at step B (e.g. weeks.generate_week_plan-style split commits at lines 206 then 208) leaves step A durably committed despite session_scope seeing the exception and 'rolling back'. The rollback only covers post-last-commit work.
  - _Risk:_ Today this is mostly latent because the MCP-exposed handlers are short and their last commit is the final statement, so partial-commit windows are tiny. The sharper latent hazard: app/api/events.py route handlers open their OWN inner `with session_scope()` for the post-commit re-fetch (events.py:145-151 etc.). Those handlers are not currently MCP-exposed — but the established convention is 'MCP wraps a REST handler,' so the first time an events-style route is added to the MCP registry it will open a nested session_scope (a second connection/transaction) while already inside one, risking it reading its own not-yet-visible writes or pinning two pool connections per call. The assistant streaming path's per-tool session_scope (api/assistant.py:309) is the right model and shows the team knows tool effects should commit independently; the REST-handler-reuse path quietly contradicts it.
  - _Recommendation:_ Document and enforce one rule: handlers reused by MCP must be flush-only and let the surrounding context own the single commit, OR MCP must not re-wrap handlers that self-commit. Concretely, either (a) split the truly-shared logic into service functions that flush only (the drafts.py/events.py service layer already does this) and have both the REST route and the MCP tool own their own commit, or (b) add a guard/assertion that a route invoked under an open session_scope never opens a nested one. At minimum, add a comment at the events.py inner-session_scope sites flagging 'do not expose this handler via MCP without refactoring the re-fetch.'
- **[LOW] Entitlement gate is check-then-act: ensure_action_allowed and increment_usage are separate, leaving a TOCTOU over-grant window**
  - _Area:_ app/services/entitlements.py:165-189 (ensure_action_allowed, read-only) vs 192-248 (increment_usage, atomic); call sites app/api/weeks.py:164+207, 439+487 and app/services/assistant_tools.py:862+901
  - _Observation:_ The quota enforcement is correctly split into a read-only pre-check (ensure_action_allowed reads the counter and raises 402 if used >= limit) and an atomic post-action increment (increment_usage's ON CONFLICT upsert). The increment itself is race-free, but the gate is not: between the check (before the expensive AI call) and the increment (after), nothing prevents N concurrent requests from all reading used < limit, all proceeding, and all incrementing. The window is wide because the AI call sits inside it.
  - _Risk:_ A free-tier user can exceed the cap by firing concurrent gated requests (e.g. several simultaneous week generations). With free limits as low as 1, a user could get 2-3x the intended free allowance. Cost/abuse impact only — no data corruption, the counter ends up accurate (it counts all the over-grant calls). Favorable-to-user direction. Genuinely a freemium-economics tradeoff, not a correctness defect.
  - _Recommendation:_ Accept as documented behavior unless abuse is observed. If you want to close it, make increment_usage atomically enforce the cap by returning the post-increment count and having callers treat new_count > limit as 'should not have run' (compensating: the AI work already happened, so this only catches the over-limit caller AFTER the fact) — or move to a check-and-increment-in-one-statement (ON CONFLICT DO UPDATE ... WHERE count < :limit, with a not-updated row signaling denial) BEFORE doing the expensive work, refunding on failure. Lowest-effort mitigation: rely on per-user request serialization at the edge for gated endpoints.
- **[LOW] Three divergent strategies for 'read fresh state after write' coexist for the same problem**
  - _Area:_ app/db.py:32 (expire_on_commit=False at the factory) vs app/api/weeks.py (session.expire_all() after every commit, ~14 sites) vs app/api/events.py:141-151 (separate session_scope re-fetch with selectinload after commit)
  - _Observation:_ The session factory deliberately sets expire_on_commit=False so ORM objects remain usable after commit without auto-refresh. weeks.py then manually calls session.expire_all() after each commit to force re-loads (re-introducing the expiry the factory turned off, globally rather than per-object). events.py rejects both and re-fetches the whole object graph in a fresh session_scope, with a comment asserting the original object is 'expired after commit' and that lazy-loading 'can race' — a claim that is at odds with expire_on_commit=False being set. So three modules solve identical post-write-hydration needs three different ways, and at least one rationale comment appears inconsistent with the configured session behavior.
  - _Risk:_ No active bug, but a coherence/maintainability hazard: a developer reasoning about post-commit object state has to know which of three regimes a given route is in. The events.py dual-session re-fetch is also the seed of the nested-session_scope hazard called out separately. Inconsistent mental models around commit/expiry are exactly what produce the harder bugs in this dimension over time.
  - _Recommendation:_ Pick one post-write hydration strategy and apply it uniformly. Given expire_on_commit=False is already the factory default, the cleanest is: after a mutating commit, explicitly session.refresh(obj, attribute_names=[...]) (or re-run the selectinload query on the same session) rather than expire_all() (which over-broadly invalidates the whole identity map) or a second session_scope (which doubles connection use and creates the nesting hazard). Reconcile or delete the events.py comment that claims objects are expired after commit, since the factory disables that.
- **[LOW] Webhook replay-dedup and notification freshness rely on check-then-insert / process-local state rather than constraint-enforced atomicity**
  - _Area:_ app/api/subscriptions.py:154-161 + 223-231 (notificationUUID dedup is SELECT-then-conditional-INSERT against ProcessedAppleNotification, PK=notification_uuid per billing.py:50-59); app/services/push_scheduler.py:38-50,161,193,258,288 (in-memory _sent_today dedup)
  - _Observation:_ Two idempotency mechanisms are correct under the assumed topology but not self-enforcing. (1) The Apple webhook dedups on notificationUUID via SELECT-then-INSERT; if Apple delivers the same notification twice concurrently, both SELECTs miss and both INSERT — the second commit raises IntegrityError on the notification_uuid primary key, which is unhandled and would 500 (Apple then retries, and the retry dedups correctly, so it self-heals, but a 500 is logged and the response is wrong for that delivery). (2) The push scheduler dedups via a module-global _sent_today dict, explicitly documented as safe only for single-instance deployment (fly.toml min_machines_running=1, decisions.md). Both are deliberate, documented choices.
  - _Risk:_ Webhook: a rare concurrent-duplicate delivery yields a logged 500 instead of a clean 204; self-healing via Apple's retry, no state corruption (the dedup PK guarantees no double-apply). Scheduler: if the app is ever scaled past one machine (auto_start_machines is true; a future scale-out or a brief two-machine overlap during deploy), _sent_today no longer dedups across processes and a user could receive the same tonight's-meal / Saturday-plan push from each instance — bounded by the APNs collapse-id, which de-dups delivery to a device but not necessarily across distinct collapse windows.
  - _Recommendation:_ Webhook: catch IntegrityError on the ProcessedAppleNotification insert (or insert it FIRST with ON CONFLICT DO NOTHING and treat 'no row inserted' as the duplicate signal) so a concurrent duplicate returns 204 instead of 500 — closing the one path where dedup degrades to an error. Scheduler: leave as-is while single-instance, but the moment scale-out is on the table, move dedup to a Postgres row (unique on (kind,user_id,date_key)) or a Postgres advisory lock, as decisions.md already anticipates.

### AI / LLM integration

_The AI layer is more mature than typical for an app this size: a clean provider-abstraction (DirectProvider/MCPProvider + per-turn ProviderAdapter), a genuinely well-engineered streaming tool loop with client-disconnect aborts, partial-text persistence, and tool-crash rollback, and a draft-first flow where every mutating tool routes through validated Pydantic payloads (DraftFromAIRequest/RecipePayload). Malformed/empty output is handled defensively almost everywhere (extract_json_object → json.loads → model_validate → RuntimeError → 502). The systemic weaknesses are not in the happy path but in three cross-cutting invariants that are missing or inconsistent: (1) output-truncation (finish_reason/stop_reason == "length"/"max_tokens") is never detected — a response cut off mid-text is silently treated as a complete, terminal answer; (2) the conversational planning assistant (the most expensive surface — up to 6 streaming turns plus tool sub-calls per message) has zero entitlement gating or per-turn metering, and no module captures token usage, so cost metering is structurally absent; (3) JSON-extraction logic is forked across two helpers with different failure modes, and the same parse/validate boilerplate is copy-pasted into ~9 modules. There is also a real prompt-injection surface: user free-text and stored profile/preference/recipe content are interpolated directly into prompts with no delimiting, and the assistant's tools mutate the user's own week — so injection is self-scoped (no cross-tenant blast radius) but can still drive unintended state changes. None of these are crashes; they are correctness/cost/robustness gaps that surface under adversarial input, long generations, and free-tier abuse._

- **[HIGH] Output truncation (finish_reason/stop_reason == length) is never detected — truncated generations are treated as complete answers**
  - _Area:_ app/services/assistant_ai.py (OpenAIAdapter.parse_stream_line, AnthropicAdapter._dispatch_pending_event, _run_provider_tool_loop), app/services/week_planner.py, app/services/recipe_search_ai.py, app/services/event_ai.py
  - _Observation:_ The streaming adapters only check whether a turn is terminal: OpenAI is_terminal=(finish_reason=="stop"), Anthropic is_terminal=(stop_reason=="end_turn"). The truncation signals (finish_reason=="length", stop_reason=="max_tokens") fall into the non-terminal else-branch. In _run_provider_tool_loop, a length-truncated turn that produced no tool calls hits `if not completed_calls:` and breaks, returning the partial accumulated_text as the final assistant_markdown with status "completed". The non-streaming JSON paths are worse: week_planner (max_tokens 8000), recipe_search (4096), event_ai/run_direct_provider (4096) parse the raw text with no stop-reason inspection at all — a JSON body truncated mid-object raises JSONDecodeError and 502s (correct but opaque), while a body truncated after a syntactically-complete-but-semantically-incomplete object parses 'successfully' into a short plan. The code comments even acknowledge truncation history ('1800 truncated structured generations ... so _parse_response raised and 502'd') but the fix was to raise max_tokens, not to detect the condition.
  - _Risk:_ On long event menus, 7-day plans, or verbose assistant replies that exceed max_tokens, the user silently receives a half-finished answer (a 4-meal week, a sentence cut off mid-word) presented as a finished, successful turn. Persisted as status=completed in AssistantMessage/AIRun, so there is no signal to retry. As models and prompts grow, this becomes more frequent and is invisible in logs.
  - _Recommendation:_ Thread the terminal *reason* (not just a bool) through NormalizedStreamEvent.turn_done. In _run_provider_tool_loop, when a turn ends with a truncation reason and no pending tool calls, either auto-continue (re-request to let the model finish) or surface an explicit 'response was cut off' state to the client and AIRun. For the non-streaming providers (run_direct_provider, week_planner._call_ai_provider, recipe_search), inspect the response's finish_reason/stop_reason before parsing and raise a distinct 'AI response truncated' RuntimeError so the route can return a retryable error instead of a silently-short plan.
- **[HIGH] Conversational planning assistant has no entitlement gating or token metering — unbounded paid LLM spend per free-tier user**
  - _Area:_ app/api/assistant.py (respond_route), app/services/assistant_ai.py (_run_provider_tool_loop), app/services/assistant_tools.py (run_tool gating), app/models/ai.py (AIRun)
  - _Observation:_ Gating is attached to discrete tool actions only: run_tool calls ensure_action_allowed/increment_usage exclusively for tools whose AssistantTool.gated_action is set (generate_week_plan→ACTION_AI_GENERATE, rebalance_day→ACTION_REBALANCE_DAY). The respond_route itself does no ensure_action_allowed — confirmed by grep finding zero entitlement references in app/api/assistant.py. But each /respond call drives _run_provider_tool_loop, which makes up to MAX_TOOL_ITERATIONS=6 streaming OpenAI/Anthropic requests (each a paid call), plus any number of read tools (get_current_week, add_meal, swap_meal, etc.) that carry gated_action=None. A free-tier user can therefore issue unlimited conversational turns — including add/swap/remove that mutate the week — without ever incrementing a counter or hitting a cap. Separately, NO module captures provider token usage: AIRun has no input/output token or cost columns, and neither the streaming loop nor run_direct_provider reads the `usage` block from provider responses. The 'metering' that exists counts API actions, not cost.
  - _Risk:_ Cost-control blast radius: an abusive or buggy client can run the planning assistant in a loop and incur unbounded OpenAI/Anthropic charges against the server key, none of which is attributable per-user (no token accounting) or capped (no per-turn gate). On the freemium model this directly undermines the paywall — the most expensive feature (multi-turn agentic planning) is free and unlimited while one-shot generate is gated.
  - _Recommendation:_ Add a turn-level gated action (e.g. ACTION_ASSISTANT_TURN) checked in respond_route before starting the worker, so conversational turns count against a free-tier budget the same way generate does. Add input_tokens/output_tokens (and optionally estimated_cost) columns to AIRun and populate them: read the `usage` object from OpenAI/Anthropic non-stream responses and the message_delta usage / final SSE usage on the streaming path. Without per-turn token capture there is no way to attribute or alert on spend.
- **[MEDIUM] User content and stored data are interpolated into prompts without delimiting — prompt-injection drives self-scoped tool mutations**
  - _Area:_ app/services/assistant_ai.py (build_planning_system_prompt, build_assistant_prompt), app/services/week_planner.py (_build_system_prompt), app/services/event_ai.py, app/services/substitution_ai.py, app/api/assistant.py (_planning_context_text)
  - _Observation:_ Every prompt builder concatenates untrusted strings directly into the instruction body with no structural separation between trusted instructions and untrusted data. Examples: request.text is appended raw as the 'Current user message'; conversation history (content_markdown) is replayed verbatim; profile_settings values (household_name, food_principles, planning_avoids…), preference signals, recipe names/notes, guest allergies/dietary_notes (event_ai _describe_guests), term aliases, and attached recipe JSON are all string-formatted into the prompt. The planning assistant additionally exposes mutating tools (add_meal, swap/confirm_swap, remove_meal, set_meal_approved, generate_week_plan, set_dietary_goal) with tool_choice=auto. There is no instruction-hierarchy hardening (no 'treat the following as data, never as instructions' fencing, no XML/delimiter wrapping). Note the multi-tenant scoping IS sound: tools resolve the week via household_id from the authenticated CurrentUser, never from model output, so an injected instruction can only act on the attacker's own household.
  - _Risk:_ A crafted recipe note, guest dietary_note, profile field, or chat message can steer the model to call mutating tools the user didn't intend (silently approve meals, wipe/replace the week via generate_week_plan, alter the dietary goal) or to ignore the allergy/avoid constraints that are injected as plain prose. Because injection is self-scoped, the damage is limited to the user's own data — but in a shared household, one member's stored content can manipulate another member's planning turn. Combined with the no-gating finding, generate_week_plan triggered by injection also spends the server AI budget.
  - _Recommendation:_ Delimit untrusted content: wrap user text, conversation history, and DB-derived context in explicit data fences and add a standing instruction that content inside fences is data, never commands. Keep allergy/avoid enforcement defense-in-depth on the server side (it partly is — merged_avoids), and consider requiring explicit client confirmation (already done for swap via the proposed_change card) for the other destructive tools (generate_week_plan, remove_meal, set_dietary_goal) so a model-only decision can't wipe state.
- **[MEDIUM] Two divergent JSON extractors plus ~9x copy-pasted parse/validate boilerplate — inconsistent robustness to the same malformed output**
  - _Area:_ app/services/assistant_ai.py (extract_json_object), app/services/week_planner.py (_extract_json), and duplicated parse blocks in recipe_search_ai.py, event_ai.py, substitution_ai.py, vision_ai.py, recipe_difficulty_ai.py, pairing_ai.py, seasonal_ai.py, recipe_drafting.py
  - _Observation:_ Two extractors with different and non-overlapping failure modes coexist. extract_json_object (shared by ~8 modules) does first-'{'-to-last-'}' substring extraction: this breaks when the model emits two JSON objects (it returns a string spanning both, which is invalid) and does not strip a leading ```json fence label. week_planner._extract_json strips markdown code fences but does NOT do brace extraction, so it breaks on any prose-wrapped JSON ('Here is your plan: {...}'). So the week-planner (a flagship feature) is MORE fragile to prose-prefixed output than the assistant, and the assistant is more fragile to multi-object output than the planner — for no principled reason. Around this, the identical try/except json.loads(...) except JSONDecodeError → RuntimeError; try model_validate except ValidationError → RuntimeError block is hand-copied into at least nine modules. recipe_drafting added a one-off retry-on-parse-failure wrapper (_provider_call_with_json_retry) that none of the other modules share.
  - _Risk:_ The same malformed model output succeeds in one feature and 502s in another, making behavior unpredictable and hard to reason about. A robustness improvement (e.g. fence stripping, balanced-brace scanning, or the parse-retry that recipe_drafting already proved useful) has to be applied nine times and will inevitably drift. The week-planner's weaker extractor means real, common LLM prose-prefixing fails the most user-visible flow.
  - _Recommendation:_ Consolidate to a single hardened parse helper — strip fences, find the first balanced top-level JSON object (not naive last-'}'), then json.loads + model_validate — and have every AI module call it. Fold the recipe_drafting one-shot retry into that helper as an optional flag. Delete week_planner._extract_json in favor of the shared one. This collapses nine copies and the two-extractor split into one consistently-robust path.
- **[MEDIUM] asyncio.run() for MCP and the day-by-day generate path bypass the loop's abort/streaming contract — partial failure isolation gaps**
  - _Area:_ app/services/assistant_ai.py (run_assistant_turn MCP branch, asyncio.run), app/services/mcp_client.py (run_codex_mcp), app/services/assistant_tools.py (_apply_week_plan_incrementally), app/api/assistant.py (worker thread)
  - _Observation:_ Two execution paths sit outside the carefully-built streaming/abort contract. (1) The MCP turn calls asyncio.run(run_codex_mcp(...)) synchronously inside the worker thread; it ignores abort_event entirely (no cancellation on client disconnect) and is a single blocking call with no incremental events, relying solely on the route's 5s heartbeat to keep the proxy alive. run_codex_mcp surfaces tool errors as a generic RuntimeError('MCP tool returned an error result') with no provider detail. The rollback fixes the DB; it cannot un-emit the SSE events the iOS client already applied.
  - _Risk:_ MCP turns can't be cancelled on disconnect (the loop spends the full provider time on a reply nobody reads — the exact problem the abort machinery was built to solve, but only wired for the direct path). For incremental generate, a failure after partial week.updated emits leaves the client UI showing meals that were rolled back server-side, requiring a manual refresh to reconcile; the client's view of the week and the DB diverge.
  - _Recommendation:_ For MCP, run run_codex_mcp on the loop with cancellation tied to abort_event (or poll abort between phases) and propagate the MCP error payload into the RuntimeError detail. - **[LOW] Anthropic stream parser silently drops tool calls whose input_json fails to parse, with no retry or surfaced error**
  - _Area:_ app/services/assistant_ai.py (AnthropicAdapter._dispatch_pending_event content_block_stop, OpenAIAdapter.parse_stream_line finish_reason branch)
  - _Observation:_ When a streamed tool call's accumulated argument JSON fails to parse, both adapters fall back to args={} (OpenAI: except JSONDecodeError → args={}; Anthropic: same, plus empty-string → {}). The call is then dispatched to run_tool with empty args. For tools with required parameters (add_meal needs day_name/meal_date/recipe_name; swap needs recipe_name; set_dietary_goal needs five fields) this produces an ok=false 'Need …' detail that is fed back to the model as a tool result. That is graceful (no crash), but a genuinely malformed/truncated tool-arg stream is indistinguishable from the model choosing to omit args, and there is no logging of the parse failure and no retry — the model has to infer from the generic error that its JSON was bad.
  - _Risk:_ Rare but real: a tool-argument stream truncated by max_tokens or corrupted produces a confusing 'Need day_name…' reply attributed to the model rather than a parse failure, with no telemetry to detect how often it happens. Combined with the truncation-not-detected finding, a length-cut tool call degrades to a vague user-facing error.
  - _Recommendation:_ Log a warning when input_json_acc/arguments is non-empty but fails json.loads (distinguishes 'model sent garbage' from 'model sent nothing'), and consider tagging the resulting tool result so the loop can ask the model to re-emit the call rather than reflecting a generic required-field error. Low effort, improves observability of a currently-invisible failure mode.

### MCP + OAuth 2.1 surface

_The OAuth 2.1 + PKCE surface is, for a single-tenant-grade meal-planning app, notably well-built: PKCE is mandatory (S256-only, no plaintext fallback), the redirect_uri allow-list rejects dangerous schemes, the authorize page HTML-escapes all attacker-influenced values, redirect params are urlencoded, codes are single-use and deleted on exchange, the MCP access-token carries aud="mcp" with HS256 pinned and require-claims enforced, and the SSO layer correctly binds an OIDC nonce into a signed, provider-scoped, purpose-scoped state JWT (M66). The cross-tenant identity hazard that stateful streamable_http would have introduced is correctly understood and defended with stateless_http + a per-request ContextVar. The systemic weaknesses are not in the crypto primitives but in the surrounding lifecycle and identity-binding invariants: scope enforcement is entirely vestigial (every tool runs with required_scopes=[] and no per-tool check), the discovery metadata is split across two implementations whose issuer/scope values can disagree and one of which (issuer) is hardcoded to a Fly hostname via a non-existent config field, the authorization "code" is pre-minted and conflated across three surfaces, there is no garbage collection or revocation path for pending requests or issued 30-day bearer tokens, and the metadata host is derived from unauthenticated X-Forwarded-* headers. None are remote-code-level, but several create durable cross-deploy and multi-tenant fragility. Overall: solid foundations, thin lifecycle/identity-invariant layer._

- **[HIGH] Scope system is declared everywhere but enforced nowhere — every token is omnipotent**
  - _Area:_ app/mcp/__init__.py (required_scopes=[]), app/mcp/auth.py, app/services/oauth.py (issue/verify), all app/mcp/*.py tools
  - _Observation:_ The whole stack advertises and threads a 'mcp' scope — metadata returns scopes_supported=['mcp'], the authorize request stores `scope`, the JWT carries a `scope` claim, JWTTokenVerifier maps it into AccessToken.scopes — yet AuthSettings is built with required_scopes=[] for both the mounted HTTP app and the standalone http path, and not a single tool in recipes.py / weeks.py / ingredients.py / profile.py / assistant.py performs any scope check. The RequireAuthMiddleware the SDK wraps the /mcp route with is therefore a pure authentication gate (valid-JWT-or-401), never an authorization gate. The `scope` plumbed end-to-end is decorative.
  - _Risk:_ Any bearer minted by the flow can invoke every tool — read recipes, mutate meal plans, delete recipes permanently (recipes_delete), and drive paid AI turns (assistant_respond, recipes_suggestion_draft). There is no way to issue a read-only or AI-disabled connection, and a client that requested a narrower scope at /authorize still receives full power because the requested scope is never narrowed against a server policy nor checked at dispatch. The day a second, less-trusted MCP consumer or a 'read-only share' feature is wanted, there is no enforcement primitive to build on, and any past-issued token silently has more authority than its scope string claims.
  - _Recommendation:_ Decide whether scopes are real. If yes: define a small scope vocabulary (e.g. mcp:read, mcp:write, mcp:ai), set required_scopes on the resource and add a per-tool scope guard (a decorator that reads the verified AccessToken.scopes from the auth context and raises 403 on mismatch) so the write/delete/AI tools are gated; intersect the DCR-/authorize-requested scope with a server allow-list before persisting it. If no: delete the scope plumbing (drop scopes_supported beyond a single marker, stop storing/claiming scope) so the surface doesn't imply an authorization model it doesn't have. Either way, the current 'advertised but unenforced' state is the dangerous middle.
- **[HIGH] Discovery metadata is split across two implementations with diverging issuer and scope values**
  - _Area:_ app/api/oauth.py (oauth_metadata, oauth_protected_resource_metadata, _public_base_url) vs app/mcp/__init__.py (AuthSettings issuer_url) and the SDK's create_protected_resource_routes
  - _Observation:_ There are two sources of truth for OAuth discovery. (1) The app's own root handlers compute every URL from `_public_base_url(request)` — the live X-Forwarded host. (2) The MCP SDK, given resource_server_url={issuer}/mcp, registers its OWN /.well-known/oauth-protected-resource/mcp route AND stamps the WWW-Authenticate `resource_metadata` challenge — both derived from `issuer`, which comes from `getattr(settings,'oauth_issuer','') or 'https://simmersmith.fly.dev'`. `oauth_issuer` is not a defined Settings field and is set nowhere, so the SDK side is permanently hardcoded to simmersmith.fly.dev. The two protected-resource handlers also disagree on scopes_supported (app handler hardcodes ['mcp']; SDK uses required_scopes=[] → empty). The app handler currently wins by router-ordering, but the SDK still owns the 401 challenge URL and the authorization_servers list it embeds.
  - _Risk:_ On any deployment whose public origin is not simmersmith.fly.dev (custom domain, preview/staging app, branch deploy, or a renamed Fly app), the 401 WWW-Authenticate challenge and the SDK-served metadata point clients at the wrong host, while the app-served metadata points at the right one. A spec-compliant MCP client that follows the RFC 9728 challenge (the canonical discovery path) gets the wrong authorization server and the connector silently fails to authenticate — a class of bug that only appears off the golden hostname, exactly where it is hardest to debug. The scopes_supported mismatch means a client reading the two documents sees contradictory capability advertisements.
  - _Recommendation:_ Collapse to one issuer source. Make the issuer request-derived to match _public_base_url (or add a real OAUTH_ISSUER config and assert it equals the public origin at startup), pass that same value into AuthSettings.issuer_url and resource_server_url so the SDK's challenge and metadata agree with the hand-rolled handlers. Reconcile scopes_supported between the two protected-resource responses. At minimum, fail fast on boot if the configured/default issuer host doesn't match the deployment, rather than silently shipping divergent metadata.
- **[MEDIUM] The authorization code is pre-minted at /authorize and reused as approval token, SSO carrier, and OAuth code**
  - _Area:_ app/services/oauth.py (create_pending_authorize_request, approve_authorize_request, exchange_code_for_token), app/api/oauth.py (authorize page, _sso_start/_sso_complete, _redirect_to_client)
  - _Observation:_ A single secret — the OAuthAuthorizeRequest primary-key `code`, minted at /authorize BEFORE any user authentication — is the authorize-page hidden form field, the query param on the Apple/Google `start` links, the value carried (indirectly, inside the signed state) through the provider hop, AND the literal `code` returned to the client's redirect_uri and exchanged at /oauth/token. There is no second authorization code generated at the moment of approval; the pre-auth identifier becomes the post-auth grant. Approval is modeled as setting user_id on that same row rather than minting a fresh, approval-bound code.
  - _Risk:_ The code's secrecy must hold from before authentication through delivery. It is rendered into an HTML page and into SSO redirect URLs (referer/proxy/log exposure surface) while still being the exact bearer of the eventual grant. If that value leaks at any pre-redirect point (browser history, a logged Location header on the deny/redirect path, an SSO URL captured by an intermediary), an attacker who also possesses or can guess nothing else still cannot pass PKCE — so this is contained by PKCE, not catastrophic — but the design couples three trust phases to one token and makes 'is this code still just a pending request vs an issued grant' a mutable-flag question (user_id null/not-null) rather than two distinct artifacts. It also means the same 32-byte value sits in logs at multiple stages (truncated to 8 chars in most, but full value in the page/redirect).
  - _Recommendation:_ Separate the pending-request identifier from the issued authorization code: keep the pre-auth row keyed by an internal id used only for the approval/SSO round-trip, and mint a fresh, short-TTL authorization code at approval time (in approve_authorize_request) that is what gets returned to the client and exchanged at /token. This restores the standard OAuth invariant that the authorization code exists only after authentication and is bound to the approved identity, and shrinks the window during which the grant-bearing secret is exposed in HTML/redirect URLs.
- **[MEDIUM] No GC for pending authorize requests and no revocation for 30-day bearer tokens**
  - _Area:_ app/models/oauth.py (OAuthAuthorizeRequest), app/services/oauth.py (exchange_code_for_token cleanup, issue/verify_mcp_access_token)
  - _Observation:_ Pending OAuthAuthorizeRequest rows are only ever deleted on two paths: a successful token exchange, or an expired-code hit inside exchange_code_for_token. Any authorize request that is started but never completed (user closes the tab, SSO abandoned, scanner/bot probing /oauth/authorize) lives in the table until — never; nothing sweeps expired-but-never-exchanged rows. There is no scheduled GC, no startup purge, no count cap. Separately, issued access tokens are stateless 30-day JWTs with no jti, no server-side record, and verify_mcp_access_token checks only signature/aud/exp — there is no revocation list, and OAuthClient has no enabled/revoked flag consulted at token-verify time.
  - _Risk:_ (1) Unbounded growth: the pending table accumulates one row per abandoned/probed authorize attempt forever; on a public endpoint this is a slow but real storage and index-bloat leak and a cheap DoS amplifier. The model docstring even claims a '60s TTL' and 'safe to garbage-collect' that no code implements (TTL is actually 1800s, GC is absent). (2) No kill switch: a leaked or mis-issued bearer is valid for a full 30 days with no way to revoke it short of rotating SIMMERSMITH_JWT_SECRET — which simultaneously invalidates every user's session JWT and every other MCP token (shared secret). Deleting/disabling an OAuthClient does not invalidate its already-issued tokens.
  - _Recommendation:_ Add a periodic (or on-token-exchange opportunistic, bulk) sweep that deletes OAuthAuthorizeRequest rows past expires_at regardless of approval state, and consider a per-client pending-row cap. For revocation, either (a) add a jti + a small revocation/issued-token table consulted in verify_mcp_access_token, or (b) at minimum check an OAuthClient.revoked flag and a per-user token-epoch at verify time so a client or user can be cut off without a global secret rotation. Fix the model docstring's stale 60s/GC claims.
- **[MEDIUM] Public metadata and challenge URLs are derived from unauthenticated X-Forwarded-* headers**
  - _Area:_ app/api/oauth.py (_public_base_url and every endpoint that builds issuer/endpoint/callback URLs from it: metadata, SSO callback URL, deny redirect)
  - _Observation:_ _public_base_url trusts X-Forwarded-Proto / X-Forwarded-Host (falling back to the Host header) with no allow-list of expected hostnames and no TrustedHostMiddleware in the app. These headers determine the issuer/authorization_endpoint/token_endpoint/registration_endpoint in the RFC 8414 document, the resource/authorization_servers in the RFC 9728 document, and — critically — the SSO callback_url that is sent to Apple/Google as redirect_uri and then echoed back. Behind Fly the edge does set these, but nothing rejects a request that arrives with attacker-chosen X-Forwarded-Host (e.g. via a second ingress path, a misconfigured proxy, or direct-to-app traffic).
  - _Risk:_ An attacker who can influence the Host/X-Forwarded-Host on a discovery or authorize request can make the server advertise authorization/token/registration endpoints on an attacker domain, or cause the SSO callback_url to point at a host the attacker controls. The provider redirect_uri allow-listing on Apple/Google's side blunts the SSO callback vector (an unregistered callback is rejected by the provider), but the metadata-poisoning vector lets a client that does discovery against a poisoned response be steered to an attacker's token endpoint. This is a classic Host-header-trust issue that is invisible in normal Fly operation and only manifests under header spoofing or a second ingress.
  - _Recommendation:_ Add Starlette TrustedHostMiddleware (or an explicit allow-list check in _public_base_url) pinning the set of valid public hostnames, and/or derive the public origin from a single configured canonical origin rather than from request headers for the security-sensitive metadata/issuer/callback URLs. Keep header-derivation only for cosmetic logging. This also dovetails with consolidating the issuer (Finding 2).
- **[LOW] MCP token lifetime and access-token expiry are asserted independently rather than derived from the JWT**
  - _Area:_ app/mcp/auth.py (JWTTokenVerifier.verify_token), app/services/oauth.py (ACCESS_TOKEN_TTL_DAYS, issue/verify)
  - _Observation:_ JWTTokenVerifier.verify_token returns AccessToken(expires_at=int(time.time()) + 30*86400) — a freshly recomputed 30-day expiry on every verification — rather than reading the verified JWT's actual `exp` claim. The JWT's true expiry is independently enforced inside verify_mcp_access_token (pyjwt checks exp), so a genuinely expired token is still rejected; but the AccessToken object the SDK propagates advertises a wrong, perpetually-30-days-out expiry, and the two TTL constants (issue side ACCESS_TOKEN_TTL_DAYS, verifier-side hardcoded 30) are coupled by convention only.
  - _Risk:_ Low blast radius today because the authoritative exp check lives in pyjwt and runs first. But the SDK's AccessToken.expires_at is now a lie: anything downstream (logging, future session caching, an SDK that decides to trust expires_at for re-auth prompts) sees a token as fresh-for-30-days when it may be seconds from its real exp. If ACCESS_TOKEN_TTL_DAYS is ever shortened, the verifier's hardcoded 30 silently disagrees, and a token near its real expiry is reported as long-lived.
  - _Recommendation:_ Have verify_mcp_access_token surface the decoded exp (and iat) and have JWTTokenVerifier set AccessToken.expires_at to the JWT's actual exp claim, removing the recomputed hardcoded constant so the SDK's view matches the cryptographic truth and the two TTL values can't drift.

### Error handling and observability

_Two high-severity systemic gaps plus two medium items._

- **[HIGH] No logging configuration: info/debug dropped, no structured logs, no error sink**
  - _Area:_ app/main.py, app/db.py, Dockerfile:24 uvicorn CMD, ~30 modules using logging.getLogger(__name__)
  - _Observation:_ Nothing calls basicConfig/dictConfig; under the default WARNING root logger all logger.info/debug are dropped and warning/exception are unformatted with no correlation. Info logs carry real signal (subscriptions.py:131, push_scheduler counts, _helpers.py:81). _RequestLogMiddleware uses bare print() because logging capture is distrusted.
  - _Risk:_ Info breadcrumbs around webhooks/push/AI/MCP never reach flyctl logs; no error-rate aggregation or alerting. Whole-service operability.
  - _Recommendation:_ Add a startup dictConfig (JSON, root INFO, stdout, propagation, correlation contextvar) plus an error sink.
- **[HIGH] Provider raise_for_status/json indexing wrapped inconsistently; unwrapped paths escape except RuntimeError as bare 500s**
  - _Area:_ week_planner.py:393,407 and :394; assistant_ai.py:744 (streaming) vs 961-968,986-992 (non-streaming) and :968; vision_ai.py:154,191 and :156; routes weeks.py:180, vision.py:66, assistant.py:507
  - _Observation:_ M6 (assistant_ai.py:963-966) wraps raise_for_status as RuntimeError; week_planner, the streaming loop, and vision_ai do not and their routes only catch RuntimeError; all four also do unguarded response.json choices/message/content indexing.
  - _Risk:_ A provider 401/429/500 or shape change throws HTTPStatusError/KeyError past except RuntimeError into a bare 500; in streaming the worker dies with no assistant.error SSE event.
  - _Recommendation:_ Apply M6 uniformly, guard json indexing, factor a shared _call_provider helper, add a global handler.
- **[MEDIUM] No global exception handler plus unhandled DB-down**
  - _Area:_ app/main.py has no add_exception_handler or app.exception_handler anywhere; contrast app/mcp/_helpers.py:71-82; app/db.py:48 get_session vs 56-67 session_scope; app/main.py:198 /api/health
  - _Observation:_ The MCP transport has a catch-all that logs and returns generic, but the REST surface has none, so uncaught route exceptions hit Starlette default handler returning generic Internal Server Error 500 logged only via uvicorn unconfigured logger. get_session only closes (no rollback/log) so a mid-request Postgres outage is an uncaught OperationalError; /api/health skips the DB.
  - _Risk:_ Unhandled errors are invisible and inconsistent; during a DB blip every request 500s unlogged with no Retry-After while health stays green, defeating Fly routing/restart.
  - _Recommendation:_ Register an Exception handler in main.py that logs method and path and returns a generic 500 with correlation id; map OperationalError to 503+Retry-After; add a deep /api/health/ready running SELECT 1.
- **[MEDIUM] Raw exception strings leak provider endpoints into responses**
  - _Area:_ assistant.py:508/518-519/545; AI service RuntimeError messages; ~30 route sites doing HTTPException detail=str(exc) (recipe_search_ai.py:203, assistant_ai.py:966, recipe_image_ai.py:203/205/242)
  - _Observation:_ Provider RuntimeErrors embed the raw httpx exception which contains the upstream URL and status. That flows into HTTPException detail and into the iOS-visible assistant.error SSE detail and the persisted message. recipe_image_ai embeds the first 200 chars of the provider body. No keys leak since httpx omits headers.
  - _Risk:_ Users and traffic capturers see provider topology, upstream URLs, and truncated error bodies; a provider body fragment can land in a chat bubble.
  - _Recommendation:_ Use standardized user-facing messages decoupled from the exception; log full server-side; never place the exception in detail or the SSE error.

### API contract consistency

_The API contract layer is solid in the mechanics that most often break mobile clients: datetime handling is uniformly timezone-aware UTC (all DB columns are DateTime(timezone=True), all defaults flow through a single tz-aware utcnow(), and no naive datetime.now() leaks into a response path), no response uses exclude_none so the iOS Codable decoder always sees explicit nulls rather than missing keys, and the dict-returning-presenter + response_model pattern is applied consistently so output is schema-validated. Request validation has thoughtful touches (SSRF guard on recipe-import URLs, length caps on free-text imports, numeric bounds on dietary goals). The weaknesses are systemic-but-non-catastrophic consistency gaps rather than correctness bugs: (1) the same upstream-AI failure class maps to two different HTTP status codes across routes, so clients cannot reliably tell "bad input" from "provider failed, retry"; (2) one persisted field (grocery source_meals) is exposed as a raw str in the week context and a list[str] in the event context; (3) state-machine fields are constrained Literals in some schemas but free str in others (week.status, event.status), which both weakens the iOS contract and creates an output-serialization 500 risk if a stored Literal value ever drifts; (4) the main library/collection endpoints (recipes, assistant threads, events) have no pagination. None of these block shipping, but they accumulate into client-side branching complexity and latent break-on-migration risk._

- **[MEDIUM] Same upstream-AI failure class maps to 422 in some routes and 502 in others**
  - _Area:_ app/api/weeks.py (181, 386, 482), app/api/recipes.py (439, 614, 712), app/api/events.py (390, 425); services/week_planner.py, recipe_drafting.py
  - _Observation:_ AI generation failures surface as RuntimeError from the service layer. week_planner.py raises RuntimeError("AI returned an invalid meal plan...") and recipe_drafting.py raises RuntimeError("AI returned invalid JSON...") — the same semantic class (the provider call failed or returned unusable output). Yet the generate-week and rebalance-day routes map RuntimeError to HTTP 422 (weeks.py:181, 482), while the side-recipe, recipe suggestion/variation/companion, pairings, cook-check, and event-menu routes map the identical RuntimeError to 502 (weeks.py:386, recipes.py:439/614/712, events.py:390/425). The status code is a per-route authoring choice, not a real distinction.
  - _Risk:_ The iOS client cannot build one reliable error-handling policy. 422 conventionally means 'your input was unprocessable — don't blindly retry'; 502 means 'upstream failed — a retry may succeed.' Because the same provider-failure produces both, the client either shows the wrong message ('check your input' for a transient OpenAI blip) or retries on a 422 that will never succeed. As AI surfaces multiply, every new endpoint re-litigates this choice and the divergence grows.
  - _Recommendation:_ Pick one mapping for upstream-AI/provider failures and apply it everywhere — 502 (or 503) for 'provider returned an error / unusable output, retry-able', reserving 422 strictly for caller-input problems (e.g. 'set a dietary goal before rebalancing', invalid meal_date). Centralize it: define a typed exception (e.g. AIProviderError vs InvalidRequestError) in the service layer and a single exception handler / small helper that maps each to its status, instead of per-route try/except str(exc) that re-decides.
- **[MEDIUM] Grocery `source_meals` exposed as `str` in week context but `list[str]` in event context**
  - _Area:_ app/schemas/week.py:200 (GroceryItemOut.source_meals: str) vs app/schemas/event.py:105 (EventGroceryItemOut.source_meals: list[str]); services/presenters.py, services/event_presenters.py, services/event_grocery.py
  - _Observation:_ The single DB column GroceryItem.source_meals (a String) is written in three different encodings depending on producer: '; '.join(...) for week aggregation (grocery.py:306), json.dumps(sorted(...)) for event aggregation (event_grocery.py:185), and the literal marker 'event:<name>' (event_grocery.py:374). It is then surfaced two contradictory ways: GroceryItemOut returns it as a raw str (iOS receives 'Monday Dinner; Tuesday Lunch'), while EventGroceryItemOut json-decodes it into list[str]. EventGroceryItemOut also omits resolution_status / overrides / retailer_prices that GroceryItemOut carries, so 'a grocery item' has two different shapes across surfaces.
  - _Risk:_ iOS must maintain two decoders and two mental models for the same domain object, and any code that tries to reuse a grocery row across the week and event surfaces will mis-parse (split-on-';' vs json-decode). If a week-context row ever holds a json-encoded value (e.g. an event-merged item that later appears in the week list), the client gets a raw JSON string where it expects delimited text — a silent display corruption, not a crash.
  - _Recommendation:_ Decide one wire representation for source_meals (a list[str] is the cleaner contract) and normalize at the presenter boundary for both surfaces, or store it structured (JSON column / separate join). Longer term, converge EventGroceryItemOut and GroceryItemOut onto a shared base so 'grocery item' is one schema with optional context-specific extensions, rather than two parallel definitions that drift independently.
- **[MEDIUM] State-machine fields are constrained Literals in some schemas, free `str` in others**
  - _Area:_ app/schemas/week.py:303 (WeekOut.status: str), app/schemas/event.py:127/138 (EventUpdateRequest.status / EventSummaryOut.status: str) vs Literal-constrained resolution_status (week.py:195, recipe.py:17), export status (week.py:159/164), and choice_mode/submission_status enums (ingredient.py)
  - _Observation:_ The codebase constrains many state fields with Literal — resolution_status ['unresolved','suggested','resolved','locked'], export destination/status, ingredient choice_mode and submission_status, event meal role. But the two most prominent lifecycle fields, week.status (values 'staging'/'ready_for_ai'/'approved' set in services/weeks.py) and event.status ('draft' default), are typed as bare str on both the response models (WeekOut/EventSummaryOut) and EventUpdateRequest. So EventUpdateRequest.status accepts any arbitrary string and persists it.
  - _Risk:_ Two-sided. (1) Contract weakness: iOS gets no enumerated guarantee for the fields that actually drive its UI state machine, so client switch statements have a silent 'unknown' fallback and a typo'd PATCH status is accepted and stored. (2) The flip side bites the strict ones: because resolution_status etc. are validated on OUTPUT (presenters return dicts that response_model re-validates), if a stored value ever drifts outside the Literal set — a future rename, a manual data fix, an older row — the GET serializes to a 500 for the whole week/recipe payload, not a graceful degrade. The two halves are the same root inconsistency: enum membership isn't owned in one place.
  - _Recommendation:_ Define each lifecycle enum once (a shared Literal/StrEnum) and use it on both the request and response models for week.status and event.status, validating EventUpdateRequest.status against it (reject unknown transitions with 422). For the output-validated Literals, either keep them strict and add a migration/back-compat test that asserts every stored value is in-set, or relax the response field to str while keeping the Literal on input — but be consistent about which direction across all state fields.
- **[MEDIUM] Core collection endpoints have no pagination**
  - _Area:_ app/api/recipes.py:171 (GET /api/recipes), app/api/assistant.py:102 (GET /threads), app/api/events.py (GET /events); services/recipes.py:list_recipes, assistant_threads.py:list_threads, events.py:list_events
  - _Observation:_ Only three endpoints bound their result set: weeks (limit, capped 24), ingredient search (limit, capped 200), nutrition search (limit, capped 50). The primary library endpoints return everything: list_recipes selects the full household library with deep selectinload chains (ingredients->base/variation, base_recipe, variants) and the presenter runs calculate_recipe_nutrition per recipe on every GET; list_threads and list_events are likewise unbounded. There is no limit/offset/cursor and no total-count envelope anywhere.
  - _Risk:_ Latency and payload size grow linearly with a household's data with no ceiling. The recipe list is the worst case — full eager-load plus per-recipe nutrition computation on every call — so a power user's library degrades the most-used screen with no server-side relief valve. Adding pagination later is a breaking change to the response shape (clients expect a bare array), so the cost only rises. There's also no consistent collection envelope ({items, next_cursor, total}) to retrofit into.
  - _Recommendation:_ Introduce a standard paginated envelope and apply it to the unbounded list endpoints, defaulting to a sane page size with an opt-in 'all' only where the client genuinely needs the full set. Even a forward-compatible limit/offset with a documented default beats unbounded. Decide the envelope shape now (even if you don't paginate every endpoint immediately) so the contract is uniform when you do.
- **[LOW] PATCH update semantics and unknown-field policy are inconsistent across resources**
  - _Area:_ app/schemas/grocery.py (GroceryItemPatchRequest, extra='forbid', null-means-clear), app/schemas/profile.py (PantryItemPatchRequest, explicit clear_* booleans), app/schemas/week.py:259 (WeekMealSidePatchRequest, extra='forbid'), app/schemas/event.py (EventMealUpdateRequest, explicit clear_* booleans, no extra policy)
  - _Observation:_ Two different idioms express 'clear this optional field' across PATCH bodies. GroceryItemPatchRequest uses Pydantic's exclude_unset/model_fields_set so 'field present + null = clear override, field absent = leave alone' (documented in the schema). But PantryItemPatchRequest and EventMealUpdateRequest instead add explicit sibling booleans (clear_typical_quantity, clear_frozen_at, clear_assignee, clear_recipe) because null already means something else there. Separately, extra='forbid' is set on only three request models (GroceryItemPatchRequest, GroceryItemQuickAddRequest, WeekMealSidePatchRequest) and absent everywhere else, so most endpoints silently drop unknown keys while a few reject them.
  - _Risk:_ Clients must remember per-endpoint which clearing convention applies — sending null to clear works on grocery items but is a no-op (or means 'no change') on pantry/event-meal fields, where a clear_* flag is required. The mixed extra policy means a typo'd field name is a silent no-op on most routes but a hard 422 on a few, so the same client bug surfaces differently depending on the resource. Both increase the surface area for 'I sent the update and nothing changed' class bugs.
  - _Recommendation:_ Standardize on one clearing convention for PATCH bodies (the exclude_unset null-means-clear idiom is the more Pydantic-native one) and document it once; reserve clear_* flags only for the genuine cases where null is itself a meaningful value. Apply a single extra policy to all request models (forbid is the safer default for a first-party client you control, since it surfaces client typos early) rather than enabling it ad hoc on three of them.
- **[LOW] Resource-creation POSTs return 200 instead of 201, and one AI route ships with no response_model**
  - _Area:_ app/api/weeks.py:118 (create_week), :267 (add_side), :576/605 (add grocery item); app/api/recipes.py:231 (save_recipe); app/api/assistant.py:110 (create_thread) — vs app/api/oauth.py:128 (the only 201) and weeks.py:355 (generate_side_recipe_route: no response_model)
  - _Observation:_ Across the API only one route declares 201 (oauth /register). Every other create-a-resource POST — create_week, add_side, add_grocery_item, save_recipe (insert path), create_thread — returns the FastAPI default 200. Separately, weeks.py:355 generate_side_recipe_route returns a bare dict draft with no response_model, so that AI-draft shape is undocumented in the OpenAPI schema and unvalidated on the way out, unlike the parallel recipe-draft routes which all declare RecipeAIDraftOut.
  - _Risk:_ Low functional impact (the iOS client keys off the body, not the status line), but it's a contract-consistency gap: the OpenAPI spec and any future REST consumer can't distinguish create from update by status, and the un-modeled side-recipe draft can silently drift from the RecipeAIDraftOut/RecipePayload shape that every other draft route guarantees — the one place a typo'd or renamed field would pass through unnoticed.
  - _Recommendation:_ Either adopt 201 consistently for resource-creating POSTs or explicitly decide 200-everywhere and document it (consistency matters more than the specific code for a single-client API). Add response_model=RecipePayload (or the appropriate draft model) to generate_side_recipe_route so its output is validated and appears in the schema like the sibling draft endpoints.

### iOS architecture

_The iOS client is built on a disciplined, modern foundation: a single `@MainActor @Observable final class AppState` partitioned cleanly into feature extensions, keychain-backed token storage with device-only/when-unlocked accessibility and explicit iCloud-sync exclusion, a thoughtful custom SSE delegate that avoids URLSession's HTTP/2 buffering, and genuinely careful concurrency hygiene around the most-touched paths (the build-87/88 migrations, scenePhase reminder sync, and the assistant stream all reason explicitly about cancellation and slot routing). Optimistic-update rollback exists where it matters most (grocery check toggle). The systemic weaknesses are at the seams: (1) there is no centralized 401/expired-token handler, so token expiry degrades to a recurring red banner instead of re-auth; (2) refresh/mutation methods freely write `syncPhase` and `lastErrorMessage` with no in-flight guard, so concurrent refreshes and per-mutation "synced" stamps race; (3) the `currentWeek`/`browsedWeek` slot model splits a single logical "displayed week" across View-local `@State` and AppState, with a fallback that silently shows the wrong week's data and routes mutations to the wrong week id when a browsed-week fetch is pending or failed; and (4) the full-tab Assistant send path, unlike the coordinator/sheet path, has no retained/cancellable task, so navigating away mid-stream leaves the server burning tokens. None are crashes; they are correctness-under-failure and consistency gaps that concentrate in exactly the multi-week / offline / token-expiry edge cases._

- **[HIGH] No centralized 401 / expired-token handling — token expiry degrades to a recurring error banner instead of re-auth**
  - _Area:_ SimmerSmithKit/API/SimmerSmithAPIClient.swift (decodeResponse, SSEStreamDelegate), App/AppState.swift + AppState+Weeks/Recipes/Assistant.swift error paths
  - _Observation:_ `SimmerSmithAPIError.unauthorized` is produced consistently for HTTP 401 in both `decodeResponse` and the SSE delegate, but no caller anywhere in the app distinguishes it from a generic error. Every catch block funnels it into `lastErrorMessage = error.localizedDescription` ("The server rejected the current bearer token.") and, for refreshes, `syncPhase = .offline/.failed`. `resetConnection()` / sign-out is only ever invoked by explicit Settings buttons (grep confirms the only call sites are SettingsView). The session JWT is long-lived but not infinite; there is no refresh-token flow.
  - _Risk:_ When the stored JWT expires or is revoked server-side, the user is not signed out and not prompted to re-authenticate. Instead every screen shows the same red "server rejected the bearer token" banner, pull-to-refresh keeps failing, the assistant stream fails, and cached content is shown as `.offline` — a confusing dead state the user can only escape by finding Settings → Sign Out. On a shared/household device this is the difference between a clean re-login and a soft-bricked app.
  - _Recommendation:_ Add one chokepoint that recognizes `.unauthorized`: either intercept it in a single `perform`/`request` wrapper in the API client (throwing a typed event), or add an `AppState.handleAuthFailure()` that the refresh/mutation catch blocks call when `error is SimmerSmithAPIError && case .unauthorized`. That handler should clear the token and drop the user to the sign-in screen (set `hasSavedConnection`-driving state) rather than painting a banner. Treat 401 categorically differently from transient network failure everywhere it can surface, including the SSE path.
- **[HIGH] syncPhase and lastErrorMessage are globally shared with no in-flight guard — concurrent refreshes and per-mutation stamps race**
  - _Area:_ App/AppState.swift (refreshAll), AppState+Weeks.swift (refreshWeek + every mutation sets syncPhase=.synced), AppState+Recipes.swift, SimmerSmithApp.swift scenePhase handler, BackgroundSyncService
  - _Observation:_ There is no `isRefreshing`/in-flight token anywhere (grep confirms none). `refreshAll()` (cold launch + sign-in + saveConnection), `refreshWeek()` (pull-to-refresh, FAB), the scenePhase `.active` reminder sync, `postClearRefreshTask`, and the build-87/88 fire-and-forget migrations can all run concurrently and all write `syncPhase` and `lastErrorMessage`. Separately, ~15 small mutation methods (`saveWeekMeals`, `approveWeek`, `regenerateGrocery`, `addMealSide`, `submitMealFeedback`, etc.) unconditionally set `syncPhase = .synced(.now)` on success even when a larger `refreshAll()` is mid-flight, and `lastErrorMessage` is a single global string that every tab and sheet writes to.
  - _Risk:_ A failing background refresh can stamp `.failed`/`.offline` over a `.synced` a user-initiated save just set (or vice-versa), so the status pill and the offline/cached UI flicker or lie. The build-88 migration explicitly works around this for itself (it bypasses the public refresh methods precisely so it won't clobber syncPhase), which is evidence the hazard is real but only patched case-by-case. `lastErrorMessage` being global means an error raised on the Recipes tab paints the banner on the Week tab (WeekView is the one view that renders it as a top overlay), and any of N concurrent operations can clear another's error by setting it to nil.
  - _Recommendation:_ Make `syncPhase` owned by exactly one refresh path: add a single `private var refreshTask: Task?` that `refreshAll`/`refreshWeek` reuse-or-coalesce, and stop having leaf mutations set `syncPhase = .synced` (they should at most clear an error, not assert global sync state). Consider scoping error surfaces per-domain (the assistant already does this well with `assistantErrorByThreadID`) rather than one shared `lastErrorMessage`, or at minimum tag errors with their origin so a stale background error can't overwrite a foreground one.
- **[HIGH] displayedWeek slot model splits one logical week across View @State + AppState and silently falls back to the wrong week**
  - _Area:_ Features/Week/WeekView.swift (displayedWeek, isViewingCurrentWeek, selectWeek, navigateRelative, the ~12 `if !isViewingCurrentWeek { appState.browsedWeek = updated }` mutation sites), App/AppState.swift browsedWeek + applyAssistantWeekUpdate
  - _Observation:_ The week the user is looking at is derived from two sources that must stay in lockstep: View-local `@State displayedWeekStart: Date?` and `appState.browsedWeek: WeekSnapshot?`. `displayedWeek` returns `appState.browsedWeek ?? appState.currentWeek` when `displayedWeekStart != nil`. Every navigation entrypoint (`selectWeek`, `navigateRelative`) sets `displayedWeekStart = target` and `browsedWeek = nil` first, then fetches into `browsedWeek` asynchronously (with `try?`/error-to-banner). AppState mutation helpers only auto-update the `currentWeek` slot; the View is responsible for manually re-routing each result into `browsedWeek` via the repeated `if !isViewingCurrentWeek { appState.browsedWeek = updated }` idiom at ~12 sites.
  - _Risk:_ Two concrete failure modes. (1) Stale-window: between setting `displayedWeekStart` and the fetch completing — or if the fetch fails (it's `try?`) — `browsedWeek` is nil, so `displayedWeek` falls back to `currentWeek`. The header label says "next week" while the body renders this week's meals, and any mutation the user makes computes `week.weekId = currentWeek.weekId`, writing edits to the WRONG week (the exact class of bug Build 104's hoist and applyAssistantWeekUpdate were created to fix — but the View-side window remains). (2) Invariant-by-convention: correctness depends on every current and future mutation site remembering to append `if !isViewingCurrentWeek { appState.browsedWeek = updated }`. One omission silently leaves the browsed view stale. `refreshWeekAfterSideMutation` centralizes this routing on the AppState side; the WeekView mutations do not, so the two layers route slots differently.
  - _Recommendation:_ Make the displayed week a single source of truth. Either fully hoist `displayedWeekStart` into AppState alongside `browsedWeek` and expose one `displayedWeek` accessor + one `applyWeekUpdate(_:)` that routes by `weekId` (so no View ever assigns `browsedWeek` directly), or have `displayedWeek` return nil/loading rather than silently falling back to `currentWeek` when the browsed slot is pending/failed (gate mutations behind a resolved displayed week). Centralize the slot-routing the way `refreshWeekAfterSideMutation` already does and delete the 12 hand-rolled reassignments.
- **[MEDIUM] Full-tab Assistant send has no cancellable task; only the coordinator/sheet path cancels in-flight streams**
  - _Area:_ Features/Assistant/AssistantView.swift (AssistantThreadView.sendMessage, button `Task { await sendMessage() }`, no onDisappear) vs Features/AIAssistant/AIAssistantCoordinator.swift (retained sendTask + cancelInFlightTurn)
  - _Observation:_ `AIAssistantCoordinator` (the sheet overlay) carefully retains `sendTask` and calls `cancelInFlightTurn()` on sheet disappear, with a documented rationale: cancellation propagates to the URLSession stream so the server sees disconnect and aborts the tool loop. The full-tab `AssistantThreadView` does the opposite: every send is fired as a bare `Task { await sendMessage() }` from button/quick-prompt taps, the task is never retained, and there is no `onDisappear`/cancellation (grep confirms no `sendTask`, no `.cancel`, no `onDisappear` in the file). The streaming session also has a 300s request / 600s resource timeout.
  - _Risk:_ If the user starts a turn in the full-tab Assistant (e.g. a long `generate_week_plan`) and navigates back / pops the NavigationStack, nothing cancels the stream. The detached `Task` keeps iterating the SSE stream and the server keeps running the LLM tool loop, burning tokens on a reply no one will read, for up to the timeout window. It also keeps mutating AppState's `assistantThreadDetails` after the view is gone. This is the same waste the coordinator path was explicitly engineered to prevent — the two assistant surfaces have divergent lifecycle discipline.
  - _Recommendation:_ Give the full-tab thread view the same treatment: retain the send `Task` in `@State` and cancel it in `.onDisappear` (or bind the send to a `.task(id:)` keyed on a send token so SwiftUI cancels it on teardown). Better, route both the sheet and the tab through the coordinator's retained-task pattern so there's one streaming-lifecycle implementation.
- **[MEDIUM] Optimistic mutations write directly to the cache with no rollback except the grocery toggle**
  - _Area:_ App/AppState+Grocery.swift (insert/replace/removeGroceryItemInCurrentWeek + saveCurrentWeek), AppState+Weeks.swift (toggleGroceryChecked is the only rollback), Features/Week/WeekView.swift mutation handlers
  - _Observation:_ `toggleGroceryChecked` is a clean optimistic-update-with-rollback: it flips `checkedGroceryItemIDs`, calls the server, and on catch restores the prior value. But it is the only one. The grocery insert/replace/remove helpers each call `try? cacheStore.saveCurrentWeek(week)` immediately, persisting local mirror edits to SwiftData. Week meal mutations are pessimistic (await server, then assign), which is safe, but they assign the returned snapshot through `currentWeek`/`browsedWeek` and `saveCurrentWeek` with the slot-routing caveat above. The cache writes themselves are `try?` — a failed `context.save()` is silently dropped.
  - _Risk:_ The asymmetry means the codebase has no general optimistic-rollback convention: a future contributor adding an optimistic local edit (the natural way to make grocery/meal edits feel instant) has only one example to copy and many counter-examples that mutate-then-persist with no inverse. Silent `try?` cache-save failures mean the in-memory `currentWeek` and the persisted cache can diverge; on next cold launch `loadCachedData()` rehydrates the stale persisted copy, surfacing edits that appear to have reverted. Low probability per-write, but it compounds across a shopping session.
  - _Recommendation:_ Establish one documented pattern for optimistic edits (capture-prior → mutate → server → rollback-on-throw) mirroring `toggleGroceryChecked`, and apply it consistently or not at all. For the cache writes, decide whether a `saveCurrentWeek` failure is recoverable; if the in-memory state is authoritative until next sync, that's fine but should be asserted in a comment, otherwise surface/log the persistence failure rather than `try?`-swallowing it (the keychain layer already moved to logging failures for exactly this reason — apply the same standard here).
- **[LOW] SSE/SwiftData error handling is pervasively try?/swallow — decode failures vanish with no telemetry**
  - _Area:_ SimmerSmithKit/Persistence/SimmerSmithCacheStore.swift (loadRecord try? decode), App/AppState.swift refreshAll (`try? await apiClient.fetchRecipeMetadata()`, threads, preferences), AppState+Assistant.swift (`event.decode` in applyAssistantStreamEvent throws but is caught generically), SimmerSmithJSONCoding date strategy
  - _Observation:_ Decoding is robust on the happy path — the custom `APIErrorResponse` decoder handles FastAPI's three error shapes, and the date strategy tries five formats before throwing. But failures are uniformly swallowed: `cacheStore.loadRecord` returns nil on any decode error (a schema-drift cache becomes silently empty), and `refreshAll` wraps several fetches in `try?` so a metadata/threads/preferences decode failure leaves those collections stale with no signal. In `applyAssistantStreamEvent`, an individual `event.decode(...)` throw propagates out of the `for try await` loop and is treated by `sendAssistantMessage` as a stream failure ("Response may be incomplete"), so one malformed frame aborts the whole turn rather than being skipped.
  - _Risk:_ When the server response shape drifts (a new required field, a renamed key, a date format the five formatters miss), the app fails closed and silent: caches read as empty, lists stop updating, or an assistant turn dies — with no logged DecodingError to diagnose against. Because `keyDecodingStrategy = .convertFromSnakeCase` is global, a server field rename produces a decode error rather than a tolerated-missing field, making the client brittle to backend evolution exactly where there's no visibility.
  - _Recommendation:_ Log DecodingErrors at the swallow points (os.Logger, as the keychain layer now does) so schema drift is diagnosable from device logs/TestFlight. Consider making per-frame SSE decode failures non-fatal (skip the bad frame, continue the stream) rather than aborting the turn. For cache loads, distinguish 'absent' from 'present-but-undecodable' so a corrupt cache can be cleared/re-fetched instead of masquerading as no data.

### Entitlements & billing

_The subscription-verification core is genuinely strong: full x5c chain validation against bundled Apple roots (not the attacker-supplied leaf), webhook idempotency via a ProcessedAppleNotification dedup table, signedDate freshness windowing, monotonic transactionId staleness rejection, terminal-status freezing of the period window, appAccountToken rebind-conflict refusal, and sandbox-acceptance gating in production. The usage-counter increment is correctly atomic (dialect-dispatched ON CONFLICT ... count+1 RETURNING). However, the *entitlement-enforcement layer wrapped around* that core has systemic gaps. Two of the four declared gated actions (pricing_fetch, recipe_import) carry free-tier limits but are enforced nowhere — a silent free-tier bypass. Entitlement state is keyed on user_id while the entire app is multi-tenant on household_id, so a paid subscription does not extend to household members and per-user counters fragment a shared household's allowance. The check-then-increment gate is non-atomic, letting concurrent free-tier requests overrun the cap. And the in_grace entitlement path is unreachable from the webhook (no DID_FAIL_TO_RENEW mapping), dropping users out of Pro during Apple's billing-retry grace window. None of these threaten the integrity of verified receipts, but several let revenue leak or wrongly revoke entitlement. Worth a focused entitlement-layer pass before relying on the freemium gate for monetization._

- **[HIGH] Two declared gated actions are never enforced — silent free-tier bypass**
  - _Area:_ app/services/entitlements.py (GATED_ACTIONS, FREE_TIER_LIMITS), app/api/recipes.py:356-406 (import-from-url/html/text), app/api/weeks.py:757-772 (pricing/import)
  - _Observation:_ GATED_ACTIONS and FREE_TIER_LIMITS declare four gated actions: ai_generate, pricing_fetch, rebalance_day, recipe_import — with concrete free-tier caps (pricing_fetch: 1, recipe_import: 5). A grep across app/ shows ensure_action_allowed / increment_usage are only ever called for ACTION_AI_GENERATE and ACTION_REBALANCE_DAY (in weeks.py and assistant_tools.py). The recipe-import routes (import-from-url/html/text) and the week pricing/import route contain no entitlement call at all — neither the gate nor the counter increment. ACTION_PRICING_FETCH and ACTION_RECIPE_IMPORT appear in source only as constant definitions and limit-table entries; they are dead as enforcement.
  - _Risk:_ Free-tier users get unlimited recipe imports and pricing imports despite the product advertising (and the admin dashboard displaying limits for) a 5-import / 1-pricing monthly cap. These flows carry real backend cost (URL fetch + parse; pricing import). The discrepancy is invisible: the admin /usage and /users endpoints will always show zero recipe_import / pricing_fetch usage because increment_usage is never called, so the gap is undetectable from telemetry. Any monetization assumption built on these caps is silently false.
  - _Recommendation:_ Either (a) wire ensure_action_allowed + increment_usage into the import and pricing routes mirroring the weeks.py:164/207 pattern (gate before work, increment after commit on success), or (b) if these are intentionally ungated, delete ACTION_PRICING_FETCH and ACTION_RECIPE_IMPORT from GATED_ACTIONS/FREE_TIER_LIMITS so the constant set is the single source of truth and the admin UI stops surfacing phantom limits. Add a test asserting every action in GATED_ACTIONS has at least one ensure_action_allowed call site (a registry-completeness invariant) to prevent future drift.
- **[HIGH] Entitlements keyed on user_id while the app is multi-tenant on household_id**
  - _Area:_ app/services/entitlements.py:124-138 (is_pro), :141-162 (counters), app/models/billing.py (Subscription PK = user_id, UsageCounter unique on user_id), app/auth.py:47-54 (CurrentUser.household_id), app/api/weeks.py:166/170 (household-scoped data)
  - _Observation:_ Subscription has user_id as its primary key (one row per user) and UsageCounter is unique on (user_id, action, period_key). is_pro() and all usage accounting resolve strictly by user_id. But the data model is multi-tenant: households support multiple members (household_members with role owner/member, invitation/accept flow in households.py), and every content route scopes by current_user.household_id. AI generation operates on household-shared weeks (load_week_or_404 uses household_id) yet bills/gates the acting user individually.
  - _Risk:_ Subscription does not follow the household. If the household owner buys Pro, a second member of the same household is still on the free tier and hits the paywall on weeks the owner's subscription paid to unlock — confusing and a support burden. Conversely, free-tier allowance is fragmented per-member rather than pooled per-household, so two free members each get the cap on the same shared week (mild revenue leak / inconsistent gating). There is no defined notion of 'this household is Pro', so entitlement semantics are ambiguous the moment a household has >1 member.
  - _Recommendation:_ Decide the entitlement unit explicitly and make it consistent end-to-end. Most natural given household-scoped content: resolve is_pro at the household level (any member with an active subscription makes the household Pro) and key UsageCounter on household_id (or (household_id, action, period)). If subscriptions are deliberately per-user, document that and gate on the acting user's own sub while still pooling free-tier usage per household. Either way, write down the invariant — the current code silently mixes both units.
- **[MEDIUM] Grace-period entitlement (in_grace) is unreachable; users lose Pro during Apple billing retry**
  - _Area:_ app/services/entitlements.py:130 (is_pro honors in_grace), app/services/subscriptions.py:244-256 (NOTIFICATION_STATUS_MAP), app/api/subscriptions.py:202-213 (webhook status application)
  - _Observation:_ is_pro() treats status in {active, in_grace} as entitled, and Subscription's docstring lists in_grace as a valid status. But NOTIFICATION_STATUS_MAP has no entry that ever produces 'in_grace'. DID_FAIL_TO_RENEW (the notification that opens Apple's billing-retry / grace window) is absent from the map entirely, so status_from_notification returns None for it and the row is left untouched. The only grace-related entry, GRACE_PERIOD_EXPIRED, maps to 'expired'. Meanwhile is_pro also requires current_period_ends_at > now (entitlements.py:138).
  - _Risk:_ When a renewal fails and Apple enters the grace/billing-retry window (during which Apple still considers the user entitled), the subscription row keeps status 'active' but its current_period_ends_at is now in the past — so is_pro returns False and a paying customer who is still entitled gets hit with the paywall. The in_grace branch that exists specifically to cover this is dead code because nothing sets that status. This wrongly revokes Pro from real subscribers during a normal Apple billing condition.
  - _Recommendation:_ Map DID_FAIL_TO_RENEW (with its WILL_NOT_AUTO_RENEW / GRACE_PERIOD subtypes) to 'in_grace', and ensure the webhook applies the renewalInfo grace-period expiry to the entitlement window so is_pro can honor it (in_grace status should not be gated out by the ends>now check, or the window should be extended to the grace deadline). Add a webhook test driving DID_FAIL_TO_RENEW and asserting is_pro stays True through the grace window.
- **[MEDIUM] Check-then-increment gate is non-atomic — concurrent free-tier requests overrun the cap**
  - _Area:_ app/services/entitlements.py:165-189 (ensure_action_allowed), :192-248 (increment_usage), app/api/weeks.py:164/207-208, app/services/assistant_tools.py:862/900-901
  - _Observation:_ The gate reads current usage and raises 402 if used >= limit (ensure_action_allowed), but the counter is only bumped later by a separate increment_usage call after the work succeeds and commits (weeks.py gates at :164, increments at :207 after a commit at :206; assistant_tools gates at :862, increments at :901 after the runner). The increment itself is atomic, but the read in the gate and the write in the increment are two separate statements with the expensive AI call between them. There is no SELECT ... FOR UPDATE, no atomic compare-and-increment, and no transactional coupling of the check to the bump.
  - _Risk:_ With a free-tier limit of N, a user firing K concurrent requests can have all K pass the gate (each reads used < limit before any increments), all do the paid AI work, and all then increment — yielding up to N+K-1 successful uses against a cap of N. For ai_generate with limit 1 this doubles or worse the free allowance per burst. It is a deliberate-abuse and accidental-double-tap vector that scales with concurrency, and the cost is the LLM spend, not just a counter discrepancy.
  - _Recommendation:_ Make the gate authoritative atomically: replace check-then-act with an atomic conditional increment (e.g. INSERT ... ON CONFLICT DO UPDATE SET count = count+1 WHERE usage_counters.count < :limit, then 402 if no row returned) performed up front, reserving the slot before the AI call; refund/decrement on failure if you want to preserve the 'only bump on success' semantics. Alternatively SELECT ... FOR UPDATE the counter row inside the request transaction. The increment_usage upsert already proves the atomic-upsert pattern is available.
- **[LOW] is_trial_pro read without a session in the profile presenter ignores the admin DB override**
  - _Area:_ app/services/presenters.py:120-121 (is_pro vs is_trial), app/services/entitlements.py:103-122 (is_trial_pro), app/services/server_settings.py (trial_mode_enabled override), app/api/admin.py:301-308 (admin trial toggle)
  - _Observation:_ is_trial_pro(settings, session=...) consults the server_settings.trial_mode_enabled DB override (admin-tunable via PATCH /api/admin/settings, build 94) when a session is supplied, and falls back to the env var otherwise. In presenters.py:120 is_pro(session, user_id) is called WITH the session (so it honors the DB trial override internally), but the very next line presenters.py:121 calls is_trial_pro() with no session — so the is_trial flag returned to the iOS client reads only the env var and ignores the admin DB toggle.
  - _Risk:_ If an operator flips trial mode on/off via the admin site (DB override) without also setting the env var, the client's is_pro behavior and the is_trial badge disagree: a user can be effectively Pro-via-trial (is_pro True) while is_trial reports False (or vice versa), so the iOS Settings screen shows the wrong copy ('you're subscribed' vs promotional trial messaging). It is presentation-only, not an access-control breach, but it makes the admin trial toggle behave inconsistently for half its surface.
  - _Recommendation:_ Pass the session: is_trial_pro(session=session) at presenters.py:121 so both is_pro and is_trial resolve trial state from the same authoritative source. Grep for other is_trial_pro() callers and apply the same; consider making the session parameter required (or defaulting the no-session path to raise in non-open mode) so the env-only path can't be reached accidentally.
- **[LOW] Admin subscription override and entitlement mutations share the legacy API token with no per-action audit identity**
  - _Area:_ app/api/admin.py:36-64 (require_admin_bearer), :431-500 (grant_pro/revoke), :255-323 (settings patch incl. trial_mode + free_tier_limits), app/main.py:157 (admin router public, self-authed)
  - _Observation:_ All admin entitlement powers — manually granting Pro until an arbitrary future date (admin_user_subscription_override grant_pro), revoking subscriptions, flipping global trial_mode_enabled, and rewriting free_tier_limits — are gated by a single shared bearer token compared against SIMMERSMITH_API_TOKEN (constant-time compare_digest, which is correct). This is the same api_token whose presence/absence also drives is_open_mode() in the entitlement gate. There is no per-operator identity, and the only audit trail for a grant is the free-text admin_note the operator optionally supplies; revoke/settings changes record no actor or timestamp of who did it.
  - _Risk:_ A single leaked shared token grants full entitlement-and-billing control (mint free Pro for any user, globally disable the paywall via trial_mode, zero out free_tier_limits). Because the same token also toggles open-mode semantics, its blast radius spans both auth posture and revenue. With no actor attribution, an incident is hard to scope or attribute. Lower severity only because the surface is operator-internal, but the concentration of powers behind one static secret is a structural risk.
  - _Recommendation:_ Treat entitlement-mutating admin endpoints as higher-privilege than read-only diagnostics: at minimum log an audit row (actor identifier from a distinct admin credential, action, target user, before/after) for grant/revoke/settings changes, and consider splitting the read-only admin token from the mutation token. Longer term, move admin auth off the shared api_token so admin access and the open-mode signal aren't coupled to the same secret.

### Client/server contract drift

_The iOS↔FastAPI contract is, on the whole, unusually disciplined and low-drift. Every endpoint the iOS client calls (95 paths) maps to a live server route; there are no orphaned client calls to removed endpoints. The wire format is internally consistent: the server uses snake_case Pydantic field names with no aliases, and the iOS decoder uses `convertFromSnakeCase`, so the two halves line up by construction. Every iOS response model is defensively coded — pervasive `decodeIfPresent(...) ?? default` and hand-written `init(from:)` decoders mean a newly-added or briefly-missing server field degrades gracefully rather than crashing the decode. The M40 week-id/slot threading is correctly implemented: `applyAssistantWeekUpdate` routes `week.updated` payloads to `currentWeek` vs `browsedWeek` by `weekId` and drops untracked weeks. Strict-typed request fields (`Literal` target_type, forbid'd PATCH bodies) are not violated by what the client actually sends. The real systemic exposure is concentrated in the assistant SSE path, where the week payload is serialized by a hand-maintained parallel encoder that bypasses the `response_model` validation gate the REST routes rely on, and where a single decode failure aborts the entire streaming turn. The Kroger teardown is effectively complete; residual references are cosmetic. Findings below are about structural fragility and dual-source-of-truth risk, not active breakage._

- **[HIGH] Assistant SSE week payload bypasses the response_model contract gate that protects every REST route**
  - _Area:_ app/api/weeks.py, app/services/presenters.py (week_payload), app/services/assistant_tools.py + assistant_ai.py (week.updated emission), app/api/assistant.py (encode_sse)
  - _Observation:_ REST week routes return the hand-built dict from week_payload() but declare response_model=WeekOut, so FastAPI re-validates/serializes the dict through WeekOut — defaults get filled, wrong types raise loudly, and the wire shape is pinned to the Pydantic schema. The SSE path is different: assistant tools and apply_ai_draft emit 'week.updated' with the SAME week_payload() dict, but encode_sse() runs it through jsonable_encoder directly with NO WeekOut filtering. So week_payload() is the single source feeding two paths, only one of which is contract-checked. Any drift in week_payload() (a renamed key, a type change, an omitted field) is caught at the REST boundary but flows silently through SSE.
  - _Risk:_ If a field that iOS decodes as required (e.g. WeekMeal.mealDate, WeekMeal.slot, WeekSnapshot.updatedAt, GroceryItem.groceryItemId) is ever dropped or mistyped in week_payload(), REST returns a loud 500 (easy to catch) while the SSE 'week.updated' frame produces a payload that fails the iOS WeekSnapshot decoder. Because applyAssistantStreamEvent is invoked with `try` inside `for try await event in stream`, that decode failure throws out of the loop and aborts the ENTIRE assistant turn — assistant.completed never applies, and the user sees 'Response may be incomplete. Pull to refresh.' One drifted field in the unguarded serializer silently breaks live AI meal-planning streaming, the product's headline flow.
  - _Recommendation:_ Route the SSE week payload through WeekOut before emitting: have week_payload() either return a WeekOut instance, or wrap emissions as WeekOut.model_validate(week_payload(...)).model_dump(mode='json') so the SSE frame is contract-identical to REST. Separately, make applyAssistantStreamEvent decode failures non-fatal per-event (catch and log/skip a single bad frame) so one malformed event can't abort a whole turn's stream. Add a round-trip test asserting jsonable_encoder(week_payload(w)) decodes cleanly as the iOS WeekSnapshot (or at minimum validates against WeekOut).
- **[MEDIUM] Two independent sources of truth for the week/meal/grocery wire shape**
  - _Area:_ app/schemas/week.py (WeekOut/WeekMealOut/GroceryItemOut/WeekMealSideOut/RetailerPriceOut) vs app/services/presenters.py (week_payload hand-built dict)

- **OAuth discovery metadata is poisonable via client-controlled Host / X-Forwarded-Host headers** — `app/api/oauth.py:56-67` [votes 0/1]
  - _Why dropped:_ All code-level facts in the finding are accurate. _public_base_url() (app/api/oauth.py:56-67) builds the origin purely from client-controlled headers (x-forwarded-proto L63, x-forwarded-host L64, host L66) with no allowlist or canonical pin. It is reflected into the RFC 8414 doc at GET /.well-known/oauth-authorization-server (L86) and the RFC 9728 doc at L95-101, and authorization_server_metadata() (app/services/oauth.py:443-454) builds authorization_endpoint/token_endpoint/registration_endpoint from that base. I confirmed there is NO TrustedHostMiddleware or host allowlist anywhere — app/main.py:156 adds only a diagnostic _RequestLogMiddleware. A canonical host literal 'https://simmersmith.fly.dev' and an oauth_issuer setting exist (app/mcp/__init__.py:150) but are unused by these endpoints. So the smell is real: these endpoints SHOULD pin to the canonical host.

But the claimed exploit does not hold against this deployment. The metadata is computed per-request from THAT request's own headers — there is no stored/shared state. Injecting X-Forwarded-Host: attacker.example only poisons the response to the attacker's OWN request; it cannot make the server hand a different victim's discovery client a poisoned document. Self-poisoning (attacker steering their own MCP client to their own AS) yields nothing. For a victim to be harmed, the hunter's own writeup concedes a precondition: a cache/CDN keying on path-but-not-Host, OR a position to inject headers into the victim's request. Neither exists here: fly.toml defines no CDN/cache layer, these JSON responses set no Cache-Control and Fly's proxy doesn't cache them, and force_https=true means an attacker who could inject X-Forwarded-Host into a victim's connection is already a TLS MITM who controls the whole response and needs no such bug. The 'Fly forwards client-supplied X-Forwarded-*' point only means the attacker's own request reflects back to the attacker — it does not cross requests/clients.

The SSO callback leg (app/api/oauth.py:464-465) is doubly defanged: it is likewise self-scoped to the attacker's own request, and the provider's redirect_uri allowlist (Apple/Google console) rejects an attacker.example callback, as the hunter concedes.

Conclusion: a genuine latent hardening gap (it should pin to oauth_issuer / the canonical host, and would become exploitable the moment a Host-agnostic cache is introduced), but not a demonstrable exploitable defect at the claimed medium severity in the as-shipped deployment. Downgrading to low (defense-in-depth) rather than not-a-bug, since the missing canonical pin is a real, fixable weakness, just not currently exploitable for cross-user credential theft.
